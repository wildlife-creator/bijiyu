# ロール・権限設計 — ビジ友（bijiyu）

## ロール体系

ビジ友のユーザーは以下の5つのロールに分類される。
1アカウントで受注者と発注者の両方の機能を利用可能（課金でアンロック）。

| ロール | 説明 | 判定方法 |
|-------|------|---------|
| 未認証 | ログイン前 | セッションなし |
| 受注者（Contractor） | 無料ユーザー | users.role = 'contractor' |
| 発注者（Client） | 有料プラン課金済み | users.role = 'client' かつ subscriptions.status IN ('active', 'past_due') |
| 担当者（Staff） | 法人プランの配下ユーザー | users.role = 'staff' かつ organization_members に所属 |
| 管理者（Admin） | サービス運営者 | users.role = 'admin' |

### ロール遷移

```
未認証 → 新規登録 → 受注者（無料）
受注者 → 課金 → 発注者（受注者機能も引き続き利用可）
発注者 → 解約 → 受注者（発注者機能がロック）
発注者 → 支払い遅延 → 猶予期間中（機能は維持）→ 猶予期間超過 → 受注者に降格
発注者（法人プラン） → 担当者を作成 → 担当者
```

### 支払い遅延（past_due）時の動作

subscriptions.status が `'past_due'`（支払い遅延 = カード決済が失敗した状態）のときの挙動:

| 項目 | 動作 |
|------|------|
| 猶予期間（グレースピリオド） | 7日間（Stripe側で設定） |
| 猶予期間中の発注者機能 | **そのまま利用可能**（ユーザー体験を損なわないため） |
| 猶予期間中の通知 | 決済失敗通知メール送信 + 管理画面にバナー表示（「お支払い方法を更新してください」） |
| 猶予期間超過時 | subscriptions.status を 'cancelled' に変更 → 受注者に降格（発注者機能ロック） |
| 掲載中の案件 | 降格時に jobs.status = 'open' の案件を 'closed'（募集終了）に変更 |
| 進行中の応募 | 既に 'accepted'（発注済み）の応募はそのまま維持（途中で止めるとトラブルになるため） |
| 担当者アカウント | 降格時に users.is_active を false に設定しログインを停止する（Middleware で is_active = false のユーザーをブロック）。組織の owner が支払いを再開し status が active に復帰した場合は、配下の担当者の is_active を true に復帰させる |

※ 猶予期間中の判定ロジック: `subscriptions.status = 'past_due'` → 発注者として扱う（'active' と同等）

## 画面アクセス権限

### 未認証ユーザー

アクセス可能: AUTH系画面（AUTH-001〜007）、静的ページ（COM-007〜011）
それ以外はすべてログイン画面にリダイレクト。

### 受注者（無料）

| カテゴリ | アクセス可能な画面 | 制限事項 |
|---------|-----------------|---------|
| マイページ | CON-001 | 発注者メニューは非表示 |
| 案件検索 | CON-002〜004 | 検索・閲覧は全案件表示。応募は登録職種かつ登録県に合致する案件のみ可能。合致しない案件は応募ボタン非活性 |
| 発注者検索 | CON-005〜006 | 制限なし |
| マイリスト | CON-007 | お気に入り案件 / お気に入り発注者の2項目 |
| メッセージ | CON-008〜010 | 新規メッセージ月5通制限。受信・返信は無制限 |
| 応募履歴 | CON-011〜013 | 自分の応募のみ |
| 空き日程 | CON-014〜016 | 制限なし |
| プロフィール | COM-001〜006 | 制限なし |
| プラン案内 | CLI-026〜027 | 閲覧・課金可能（課金導線のため受注者にも開放。CLI系だが例外的にアクセス許可） |
| 発注者系 | CLI-001〜025 | **アクセス不可**（CLI-026〜027 は上記の通り例外） |
| 管理者系 | ADM-001〜015 | **アクセス不可** |

### 発注者（課金済み）

受注者の全機能に加えて、以下が解放される。

| カテゴリ | アクセス可能な画面 | 備考 |
|---------|-----------------|------|
| マイページ | CON-001 | 発注者メニューが追加表示 |
| マイリスト | CON-007 | 見込みユーザー一覧が追加（3項目） |
| 募集現場 | CLI-001〜004 | プランにより掲載件数制限あり |
| 職人検索 | CLI-005〜006 | 無制限 |
| 応募管理 | CLI-007〜009 | 制限なし |
| 発注履歴 | CLI-010〜012 | 制限なし |
| メッセージ | CLI-013〜015 | 一斉送信・スカウト送信が可能 |
| テンプレート | CLI-016〜019 | 制限なし |
| 発注者情報 | CLI-020〜021 | 自社情報のみ |
| 担当者管理 | CLI-022〜025 | 法人プランのみ |
| 管理者系 | ADM-001〜015 | **アクセス不可** |

### 管理者

一般ユーザー画面にはアクセス不可。管理者系画面（ADM-001〜015）のみ。

| カテゴリ | アクセス可能な画面 | 備考 |
|---------|-----------------|------|
| 管理者認証 | ADM-001 | 一般ユーザーとは別のログイン画面 |
| 管理者トップ | ADM-002 | 全管理機能への導線 |
| 発注者管理 | ADM-003〜007 | 発注者アカウントの閲覧・編集・作成 |
| ユーザー管理 | ADM-008〜010 | ユーザーの閲覧・動画代理投稿 |
| 本人確認 | ADM-011〜012 | 承認・否認（メール通知 + ステータス更新） |
| 応募履歴 | ADM-013〜014 | 全応募の閲覧・CSV出力 |
| パスワード | ADM-015 | 自分のパスワード変更のみ |

## 法人プランの組織権限

法人プラン（¥48,000 / ¥148,000）では、1つの発注者アカウントの下に複数の担当者を配置できる。

### 用語の区別（重要）

「管理者」という言葉が2つの意味で使われるため、以下のように区別する:

| 用語 | 英語表記 | 意味 | 該当テーブル |
|------|---------|------|------------|
| **システム管理者** | Admin | ビジ友の運営チーム。ADM画面を操作する人 | users.role = 'admin' |
| **組織管理者** | Org Admin | 法人プランで担当者の追加・編集ができる権限を持つ人 | organization_members.org_role = 'admin' |
| **管理責任者** | Org Owner | 法人プランの契約者本人。組織の最上位権限 | organization_members.org_role = 'owner' |

※ コード内では `isSystemAdmin()` と `isOrgAdmin()` のように関数名で明示的に区別する。

### 権限階層

```
管理責任者（Org Owner）
├── 組織管理者（Org Admin）
│   └── 担当者（Staff）
└── 組織管理者（Org Admin）
    └── 担当者（Staff）
```

### 操作権限マトリクス

| 操作 | 管理責任者 | 管理者 | 担当者 |
|------|----------|--------|-------|
| 募集現場の作成/編集/削除 | ○ | ○ | ○ |
| メッセージ送信 | ○ | ○ | ○ |
| スカウト送信 | ○ | ○ | ○ |
| 応募の発注可否判断 | ○ | ○ | ○ |
| 担当者の作成/編集/削除 | ○ | ○ | — |
| 管理者の作成 | ○ | — | — |
| 管理責任者の変更 | 内部管理者のみ | — | — |
| プラン変更 | ○ | — | — |
| 代理アカウント設定 | ○ | ○ | — |
| 退会 | ○ | — | — |

### 代理アカウント

- 用途: 運営スタッフが発注者の代わりにメッセージを送信するためのアカウント
- 権限: 担当者と同じ
- 制約: 1法人につき1つのみ作成可能（法人プランのみ）
- 設定: 担当者新規作成/編集画面のチェックボックスで指定

## メッセージ送信制限の実装方針

| ユーザー種別 | 制限内容 | 制御方法 |
|------------|---------|---------|
| 無料ユーザー | 新規メッセージ月5通 | サーバー側でカウント（RLS or Server Action） |
| 有料ユーザー | 無制限 | 制限なし |
| 一斉送信 | 発注者のみ | RLSで制御 |
| 代理メッセージ通数 | 制限なし | システムでは制御しない（運用管理） |

※ 「新規メッセージ」= まだやり取りしたことがない相手への初回メッセージ。返信は制限に含まない。

## 権限チェックの実装パターン

### 三重防御（Middleware + Server Action + RLS）

```
リクエスト
  → Middleware（第1層: ルーティング制御 = どのページにアクセスできるか）
    → Server Action（第2層: ビジネスロジック = その操作が許可されているか）
      → Supabase RLS（第3層: データアクセス制御 = どの行を読み書きできるか）
```

**Middleware（第1層）で判定する内容:**
- 未認証 → `/auth/*` 以外をリダイレクト
- 受注者 → 発注者専用画面と管理者画面をブロック（※ CLI-026〜027 は例外で許可）
- 発注者 → 管理者画面をブロック
- 管理者（システム管理者） → `/admin/*` のみアクセス可

**Server Action（第2層）で判定する内容:**
- 認証チェック（ログイン済みか）
- ロールチェック（その操作が許可されたロールか）
- 課金チェック（有料機能に無料ユーザーがアクセスしていないか）
- 所有権チェック（自分のデータに対する操作か）
- ビジネスルール（無料ユーザーの応募制限、メッセージ月5通制限、レート制限など）
- 詳細は security.md の「Server Actions の権限チェック」セクションを参照

**RLS（第3層）で判定する内容:**
- 自分のデータのみ読み書き可（基本原則）
- 公開プロフィールは全ユーザーが閲覧可
- メッセージは送信者・受信者のみ閲覧可
- 本人確認書類は本人 + システム管理者のみ
- 案件は作成者が編集可、全ユーザーが閲覧可
- 詳細は database-schema.md の「主要テーブル別ポリシー」セクションを参照
