# ロール・権限設計 — ビジ友（bijiyu）

## ロール体系

ビジ友のユーザーは以下の5つのロールに分類される。
1アカウントで受注者と発注者の両方の機能を利用可能（課金でアンロック）。

| ロール | 説明 | 判定方法 |
|-------|------|---------|
| 未認証 | ログイン前 | セッションなし |
| 受注者（Contractor） | 無料ユーザー | users.role = 'contractor' |
| 発注者（Client） | 有料プラン課金済み | users.role = 'client' かつ subscriptions.status IN ('active', 'past_due') |
| 担当者（Staff） | 法人プランの配下ユーザー | users.role = 'staff' かつ organization_members に所属 |
| 管理者（Admin） | サービス運営者 | users.role = 'admin' |

### ロール遷移

```
未認証 → 新規登録 → 受注者（無料）
受注者 → 課金 → 発注者（受注者機能も引き続き利用可）
発注者 → 解約 → 受注者（発注者機能がロック）
発注者 → 支払い遅延 → 猶予期間中（機能は維持）→ 猶予期間超過 → 受注者に降格
発注者（法人プラン） → 担当者を作成 → 担当者
```

### 担当者（Staff）と受注者アクションの制限

担当者（`users.role = 'staff'`。`org_role` が `admin` でも `staff` でも同様）は発注者（法人プラン）が作成するアカウントであり、自分で新規登録→プロフィール入力のフローを経ていない。

**CON系画面の閲覧は可能だが、受注者としてのアクション（応募・空き日程管理等）は不可。**

閲覧を許可する理由:
- 発注業務のための市場調査（競合の募集条件・報酬相場の確認）に活用できる
- 取引先の発注者情報を確認できる

受注者アクションを制限する理由:
1. **アカウントのライフサイクル問題**: 担当者のアカウントは組織に依存する（組織のサブスク解約 → is_active = false）。もし応募中に組織のサブスクが切れると、応募が宙に浮き、相手方の発注者にも悪影響が出る
2. **プロフィール未登録**: 担当者は名前・メール・権限のみで作成されるため、受注者として必要な情報（職種・スキル・資格・対応エリア等）が揃っていない
3. **ライフサイクルの非自律性**: アカウントの有効/無効が組織オーナーの判断に依存し、本人の意志と無関係に受注活動が停止されるリスクがある

**担当者が受注機能を使いたい場合**: 別途個人の無料アカウント（別メールアドレス）を作成して利用する。個人アカウントはライフサイクルが独立しているため、上記の問題が発生しない。

**CON系画面ごとの担当者アクセスルール:**

| 画面 | 閲覧 | 制限内容 |
|------|-----|---------|
| CON-001 マイページ | ○ | 「仕事を探す」セクションも表示（閲覧導線として） |
| CON-002 募集案件一覧 | ○ | 閲覧のみ（UIは全ロール共通） |
| CON-003 募集案件詳細 | ○ | **応募ボタン非表示**（staffは応募不可） |
| CON-004 応募情報入力 | × | Middlewareでルートブロック（応募フォーム自体にアクセス不可） |
| CON-005 発注者一覧 | ○ | 閲覧のみ |
| CON-006 発注者詳細 | ○ | 閲覧のみ |
| CON-007 マイリスト | ○ | 案件・発注者のお気に入りは可能（市場調査の参考情報として活用） |
| CON-008〜010 メッセージ | ○ | 組織スレッドとして既に共有済み。個人としての新規スレッド作成は不可 |
| CON-011〜013 応募履歴 | × | Middlewareでルートブロック（staffは応募できないのでデータが存在しない） |
| CON-014〜016 空き日程 | × | Middlewareでルートブロック（個人の受注者情報であり、staffには不要） |

### 支払い遅延（past_due）時の動作

subscriptions.status が `'past_due'`（支払い遅延 = カード決済が失敗した状態）のときの挙動:

| 項目 | 動作 |
|------|------|
| 猶予期間（グレースピリオド） | 7日間（Stripe側で設定） |
| 猶予期間中の発注者機能 | **そのまま利用可能**（ユーザー体験を損なわないため） |
| 猶予期間中の通知 | 決済失敗通知メール送信 + 管理画面にバナー表示（「お支払い方法を更新してください」） |
| 猶予期間超過時 | subscriptions.status を 'cancelled' に変更 → 受注者に降格（発注者機能ロック） |
| 掲載中の案件 | 降格時に jobs.status = 'open' の案件を 'closed'（募集終了）に変更 |
| 進行中の応募 | 既に 'accepted'（発注済み）の応募はそのまま維持（途中で止めるとトラブルになるため） |
| 担当者アカウント | 降格時に users.is_active を false に設定しログインを停止する（Middleware で is_active = false のユーザーをブロック）。組織の owner が支払いを再開し status が active に復帰した場合は、配下の担当者の is_active を true に復帰させる |
| past_due 中の解約 | **許可**（即時実行）。アップグレード・ダウングレードは不可。前提条件チェック（掲載中案件・未返信応募・担当者数）はスキップし、案件は強制クローズ。警告ダイアログで即時効果を事前告知する |
| past_due 中のプラン変更 | **不可**。CLI-026 でアップグレード・ダウングレードボタンを非活性化 |

※ 猶予期間中の判定ロジック: `subscriptions.status = 'past_due'` → 発注者として扱う（'active' と同等）

### ダウングレード・解約の前提条件

プラン変更（ダウングレード）または解約を実行する前に、以下の3項目をチェックする。条件を満たさない場合はブロック:

| チェック | 条件 | エラーメッセージ |
|---------|------|----------------|
| 掲載中案件数 | ダウングレード先の上限以下 | 「掲載中の案件を{上限}件以下にしてください（現在{N}件）」 |
| 未返信の応募 | 0件 | 「未対応の応募があります。すべて対応してからプラン変更してください」 |
| 担当者数 | ダウングレード先の上限以下 | 「担当者を{上限}人以下にしてください（現在{N}人）」 |

※ 発注済み（accepted）の応募はブロック対象外。解約後もメッセージスレッドは利用可能（無料プランの月5スレッド制限が適用）
※ 組織データはダウングレード・解約時も削除しない（再アップグレード時に再利用）
※ past_due 中の解約はこれらのチェックをスキップする（猶予期間超過の自動解約と同等の処理）
※ 詳細は `.kiro/specs/billing/requirements.md` REQ-BL-005 を参照

## 法人プランの組織権限

法人プラン（¥48,000 / ¥148,000）では、1つの発注者アカウントの下に複数の担当者を配置できる。

### 用語の区別（重要）

「管理者」という言葉が2つの意味で使われるため、以下のように区別する:

| 用語 | 英語表記 | 意味 | 該当テーブル |
|------|---------|------|------------|
| **システム管理者** | Admin | ビジ友の運営チーム。ADM画面を操作する人 | users.role = 'admin' |
| **組織管理者** | Org Admin | 法人プランで担当者の追加・編集ができる権限を持つ人 | organization_members.org_role = 'admin' |
| **管理責任者** | Org Owner | 法人プランの契約者本人。組織の最上位権限 | organization_members.org_role = 'owner' |

※ コード内では `isSystemAdmin()` と `isOrgAdmin()` のように関数名で明示的に区別する。

### 権限階層

```
管理責任者（Org Owner）
├── 組織管理者（Org Admin）
│   └── 担当者（Staff）
└── 組織管理者（Org Admin）
    └── 担当者（Staff）
```

### 代理アカウント

- 用途: **ビジ友の運営スタッフ**（サービス提供側の人間）が、法人の発注者に代わってメッセージ対応を行うためのアカウント
- 仕組み: 法人組織内に担当者アカウントとして作成し、ビジ友の運営スタッフがそのアカウントにログインして操作する。システム上は通常の担当者（`users.role = 'staff'`）と同じ権限で動作する。`is_proxy_account = true` フラグにより「社内の実際の従業員」ではなく「外部の運営者が操作するアカウント」であることを識別できる
- メッセージ送信: 通常の担当者と同じフローで送信する。sender_id の書き換えは行わない。Server Action が送信者の `is_proxy_account` を参照し、`messages.is_proxy = true` を自動設定する
- 「代理」バッジ: 発注者側の画面でのみ表示（受注者側には表示しない。受注者には企業名のみ表示される）
- 通数上限: プラン説明上の目安（法人プラン: 年36通、高サポート: 年300通）。システムでは制御しない（運用管理）
- 権限: 担当者と同じ（特別な権限チェックは不要）
- 制約: 1法人につき1つのみ作成可能（法人プランのみ）
- 設定: 担当者新規作成/編集画面（CLI-024/025）の「代理アカウント」チェックボックスで指定

## 権限チェックの実装パターン

### 三重防御（Middleware + Server Action + RLS）

```
リクエスト
  → Middleware（第1層: ルーティング制御 = どのページにアクセスできるか）
    → Server Action（第2層: ビジネスロジック = その操作が許可されているか）
      → Supabase RLS（第3層: データアクセス制御 = どの行を読み書きできるか）
```

**Middleware（第1層）で判定する内容:**
- 未認証 → `/auth/*` 以外をリダイレクト
- 受注者 → 発注者専用画面と管理者画面をブロック（※ CLI-026〜027 は例外で許可）
- 発注者 → 管理者画面をブロック
- 管理者（システム管理者） → `/admin/*` のみアクセス可

**Server Action（第2層)で判定する内容:**
- 認証チェック（ログイン済みか）
- ロールチェック（その操作が許可されたロールか）
- 課金チェック（有料機能に無料ユーザーがアクセスしていないか）
- 所有権チェック（自分のデータに対する操作か）
- ビジネスルール（無料ユーザーの応募制限、メッセージ月5通制限、レート制限など）
- 詳細は security.md の「Server Actions の権限チェック」セクションを参照

### 所有権チェックと組織メンバー判定（必ず守ること）

**法人プラン機能**で「特定のリソース（案件・応募等）に対する操作権限」を判定する際、`owner_id === user.id` だけで判定してはならない。組織メンバーが作成したリソースも、同じ組織のオーナーや他のメンバーが操作できる必要があるため。

**正しい判定パターン:**

```typescript
// 1. owner_id === user.id を最初にチェック（個人プラン or 自分の作成物）
let authorized = resource.owner_id === user.id;

// 2. 満たさなければ組織メンバーシップをチェック
if (!authorized && resource.organization_id) {
  const { data: orgMember } = await admin
    .from("organization_members")
    .select("organization_id")
    .eq("user_id", user.id)
    .eq("organization_id", resource.organization_id)
    .maybeSingle();
  authorized = !!orgMember;
}

// 3. どちらも満たさなければ拒否
if (!authorized) return { success: false, error: "..." };
```

**UI 表示範囲との整合（必ず守ること）:**

プルダウン / リストで「操作可能なリソース」として提示した範囲と、実際の Server Action での許可範囲は**必ず一致**させる。

- 禁止パターン: UI では組織全体の案件を選択できるのに、Server Action は `owner_id === user.id` のみ許可 → プルダウンで選べるのに購入できない UX バグ
- 正しいパターン: UI のクエリも Server Action の権限判定も、どちらも「オーナー OR 組織メンバー」の共通ロジックで判定する

**過去に発生した例:**

急募オプション購入（billing）で、プルダウンは組織全体の案件を表示するのに Server Action は `owner_id` のみで判定していたため、スタッフ作成案件を選ぶと「対象の案件が見つからないか、操作する権限がありません」エラーになった。

**RLS（第3層）で判定する内容:**
- 自分のデータのみ読み書き可（基本原則）
