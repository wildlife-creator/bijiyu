# セキュリティ方針 — ビジ友

## 基本原則
- 多層防御（Defense in Depth）: Middleware → Server Action → RLS の三重防御（詳細は「Server Actions の権限チェック」セクション参照）
- 最小権限（Least Privilege）: デフォルトで拒否、明示的に許可
- セキュアバイデフォルト: 新機能は制限付きで開始

## 入力検証・サニタイズ

### フォーム入力
- Zod スキーマによるバリデーション（クライアント + サーバーの両方で実施）
- 許可リスト方式を優先（ブロックリストではなく）
- エラーメッセージに内部情報を含めない

### ファイルアップロード
- 許可するファイルタイプを明示的に制限
  - プロフィール画像: JPEG, PNG（最大5MB）
  - 現場写真: JPEG, PNG（最大10MB）
  - 本人確認書類: JPEG, PNG, PDF（最大10MB）
  - メッセージ添付画像: JPEG, PNG（最大10MB）
- MIMEタイプ（= ファイルの種類を表す識別子）とファイル拡張子の両方を検証（片方だけだと偽装される恐れがある）
- アップロード先: Supabase Storage（バケットごとにアクセス制御）
- ファイル名はUUID（ランダムな一意の文字列）に変換（元のファイル名を保存しない = ファイル名からの情報漏洩を防止）
- 画像はサーバー側でリサイズ・最適化を推奨（帯域の節約とパフォーマンス向上）

## 機密データの保護

### 本人確認書類（最重要）
- 保管場所: Supabase Storage の非公開バケット（`identity-documents`）
- アクセス制御:
  - 本人: 自分の書類のみ閲覧可（RLS）
  - 管理者: 承認フロー時のみ閲覧可（RLS）
  - 他のユーザー: アクセス不可
- 承認完了後の扱い:
  - 承認済みフラグをDBに記録
  - 書類の原本の保持期間: **法令確認後に確定する（MVP開発着手前に決定を推奨）**
    - 確認すべきポイント: 個人情報保護法における本人確認書類の保管義務期間、建設業法上の要件
    - 確定するまでは書類を削除しない（後から「保管すべきだった」となるリスクを回避）
    - 確定後、Edge Function 等で自動削除の仕組みを実装する
- ログ: 書類へのアクセスはすべて監査ログに記録

### 個人情報（PII）
- メールアドレス: 部分マスクで表示（例: t***@gmail.com）を他ユーザー向けに適用
- 電話番号: プロフィールに含めない（メッセージ機能で代替）
- 住所: 都道府県レベルまでの公開に制限

### 決済情報
- カード情報はStripeが管理（サーバー・DBに一切保存しない）
- Stripe Customer IDのみDBに保持

## Supabase Storage バケット設計

| バケット名 | 用途 | アクセスレベル |
|-----------|------|-------------|
| avatars | プロフィール画像 | 公開（認証不要で閲覧可） |
| job-attachments | 案件の現場写真・書類 | 公開（認証不要で閲覧可） |
| identity-documents | 本人確認書類 | 非公開（本人 + システム管理者のみ） |
| ccus-documents | CCUS書類 | 非公開（本人 + システム管理者のみ） |
| message-attachments | メッセージ添付画像 | 非公開（スレッド参加者のみ） |

### Storage RLS ポリシー（= 誰がどのファイルを読み書きできるかのルール）

Supabase Storage にもテーブルと同様に RLS を設定できる。
ファイルパスにユーザーIDを含めることで「自分のファイルだけ操作可能」というルールを実現する。

| バケット | 読む（SELECT） | アップロード（INSERT） | 削除（DELETE） |
|---------|-------------|-------------------|-------------|
| avatars | 誰でも可 | 本人のみ（パスに自分のIDが含まれるファイルのみ） | 本人のみ |
| job-attachments | 誰でも可 | 案件の作成者または同一組織メンバー | 案件の作成者または同一組織メンバー |
| identity-documents | 本人 + システム管理者のみ | 本人のみ | 本人のみ |
| ccus-documents | 本人 + システム管理者のみ | 本人のみ | 本人のみ |
| message-attachments | スレッド参加者のみ | スレッド参加者のみ | 不可（送信済みメッセージの添付は消せない） |

### Storage のファイル保存先パス（フォルダ構造のルール）

ファイルパスにユーザーIDや案件IDを含めて整理する。これにより RLS の判定がシンプルになる。

```
avatars/{user_id}/{ファイル名}.jpg
job-attachments/{job_id}/{ファイル名}.jpg
identity-documents/{user_id}/{書類種別}_{タイムスタンプ}.jpg
ccus-documents/{user_id}/{CCUS技能者ID}_{タイムスタンプ}.jpg
message-attachments/{thread_id}/{message_id}_{ファイル名}.jpg
```

## メッセージ機能のセキュリティ

### スパム・悪用防止
- 無料ユーザー: 月5通の送信制限（サーバー側でカウント、クライアント側のみの制限は不可）
- 一斉送信: 発注者プランのみ許可（RLSで制御）
- レート制限: 同一ユーザーからの連続送信を制限（1分間に3通まで）

### レート制限の実装方針

レート制限（= 短時間に大量の操作を繰り返すのを防ぐ仕組み）は以下の方法で実装する:

| 層 | 方法 | 説明 |
|----|------|------|
| Server Action（第1層） | メモリ内カウンター or Supabaseクエリ | メッセージ送信 Server Action の冒頭で、直近1分間の送信数を messages テーブルから COUNT して判定。3通以上ならエラーを返す |
| フロントエンド（補助） | 送信ボタンの無効化 | 送信後に一定時間ボタンを押せなくする（UX改善目的。セキュリティ的にはサーバー側が本体） |

具体的な判定クエリ:
```sql
-- 直近1分間のメッセージ送信数を確認
SELECT COUNT(*) FROM messages
WHERE sender_id = {現在のユーザーID}
  AND created_at > NOW() - INTERVAL '1 minute';
-- 結果が 3 以上なら送信を拒否
```

※ 月5通の新規メッセージ制限は、新規スレッド作成数（message_threads の当月作成数）でカウントする。返信は制限に含まない。

**初期段階の実装方針:**
- レート制限（1分間3通）も月次制限（月5通）も、**データベースへの COUNT クエリで判定する**（Redis 等の外部キャッシュは使わない）
- messages テーブルの (sender_id, created_at) インデックス（検索を速くするための仕組み）が既に設定されているため、COUNT クエリは十分高速
- **将来的な改善**: ユーザー数が増えてクエリが遅くなった場合のみ、Redis（高速キャッシュ = データを一時的にメモリに記憶して素早く読み書きできる仕組み）に移行を検討する。Supabase の Query Insights（クエリ分析ツール）で遅いクエリが見つかった場合に判断する

### 応募制限（ビジネスロジックバリデーション）
- 無料ユーザーの応募時、Server Actionで以下を検証:
  - 案件の職種が申請者の登録職種に含まれるか
  - 案件の都道府県が申請者の対応可能エリアに含まれるか
- フロントエンド: 合致しない案件は応募ボタンを非活性にし、メッセージを表示
- サーバーサイド: フロントエンドのバイパス対策として、Server Actionでも同じ検証を実施
- 有料ユーザーにはこの制限を適用しない

### コンテンツセキュリティ
- メッセージ内のURLはサニタイズ（XSS防止）
- 外部URL表示時は警告を表示

## Server Actions の権限チェック

Next.js の Server Actions（サーバーアクション = フォーム送信やボタン押下時にサーバー側で実行される処理）は、Middleware（ミドルウェア = リクエストの関所）を経由せず直接呼び出される可能性がある。
そのため、**すべての Server Action の冒頭で必ず権限チェックを行う**。

### 実装パターン（三重防御）

```
リクエスト
  → Middleware（第1層: ルーティング制御。ページアクセスの粗いチェック）
    → Server Action（第2層: ビジネスロジックの権限チェック）
      → Supabase RLS（第3層: データベースの行単位アクセス制御）
```

### Server Action の必須チェック項目

| チェック項目 | 説明 | 例 |
|------------|------|-----|
| 認証チェック | ログイン済みかどうか | `const user = await getUser(); if (!user) throw new Error('Unauthorized');` |
| ロールチェック | その操作が許可されたロールかどうか | 発注者専用機能 → `if (user.role !== 'client') throw ...` |
| 課金チェック | 有料機能に無料ユーザーがアクセスしていないか | `if (!isPaidUser(user.id)) throw ...` |
| 所有権チェック | 自分のデータに対する操作か | `if (job.owner_id !== user.id) throw ...` |
| ビジネスルールチェック | 業務上の制約を満たしているか | 無料ユーザーの応募 → 職種・エリア一致確認 |

### Server Action で制御すべき主なビジネスロジック

- 無料ユーザーの応募制限（登録職種 × 登録県の一致チェック）
- 無料ユーザーの新規メッセージ月5通制限
- メッセージ送信のレート制限（1分間に3通まで）
- 案件掲載件数制限（個人プラン = 1件/月）
- 担当者の作成権限チェック（管理者以上のみ）
- 代理送信の権限チェック（法人プランの代理アカウントのみ）

## メール送信失敗時の共通方針

全 spec 共通のルールとして、メール送信失敗時は以下のように処理する:

- **原則**: メール送信失敗で本体処理をロールバック（取り消し）しない
- **理由**: メール送信は外部サービス（Resend）に依存するため、一時的なネットワーク障害で本体処理まで巻き戻すとデータの不整合が生じる
- **処理**: 送信失敗をログに記録する（console.error + 必要に応じて audit_logs）
- **リトライ**: Resend の自動リトライ機能に委任する。アプリ側での手動リトライは初期段階では実装しない
- **適用箇所**: auth（登録完了通知）、admin（アカウント作成通知）、organization（担当者作成通知）、billing（支払い失敗通知・解約通知）、matching（発注/お断り通知）の全メール送信処理

## HTTPセキュリティヘッダー

Next.js Middleware または next.config.js で以下を設定:

| ヘッダー | 値 | 目的 |
|---------|-----|------|
| Strict-Transport-Security | max-age=31536000; includeSubDomains | HTTPS強制 |
| X-Content-Type-Options | nosniff | MIMEタイプスニッフィング防止 |
| X-Frame-Options | DENY | クリックジャッキング防止 |
| Content-Security-Policy | default-src 'self'; ... | XSS防止 |
| Referrer-Policy | strict-origin-when-cross-origin | リファラー漏洩防止 |

## 環境変数・シークレット管理
- Supabase URL / キー: Vercel環境変数
- Stripe秘密鍵 / Webhook署名鍵: Vercel環境変数
- Resend APIキー: Vercel環境変数
- コードへのハードコード: 絶対禁止
- .env.local はGitで管理しない（.gitignoreに含める）

## 監査ログ
- 記録すべきイベント:
  - **認証系**: ログイン成功/失敗（auth）
  - **本人確認系**: 本人確認書類へのアクセス、承認/否認操作（admin REQ-ADM-012）
  - **アカウント系**: アカウント削除（ソフトデリート）、担当者の作成/削除（organization）
  - **課金系**: ロール変更（課金によるアップグレード/ダウングレード）、Webhook 処理結果（billing）、サブスクリプションステータス変更、past_due 自動解約
  - **メッセージ系**: 代理送信（messaging — sender_id と proxy_sender_id の両方を記録）
  - **管理者操作**: 管理画面での全操作（admin）
- 記録フォーマット: audit_logs テーブルの action カラムに上記のイベント種別を設定。metadata（jsonb）に変更前後の値や補足情報を含める
- ログにパスワード・トークン・機密情報を含めない
