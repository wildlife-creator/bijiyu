# プロフィール機能（profile）— 要件定義

## 概要

ユーザーのプロフィール閲覧・編集、本人確認・CCUS登録申請、退会手続きを提供する機能。

## 対象画面

| 画面ID | 画面名 | 概要 |
|--------|--------|------|
| COM-001 | プロフィール詳細 | プロフィール情報の表示 |
| COM-002 | プロフィール編集 | プロフィール情報の編集 |
| COM-003 | 本人確認・CCUS登録申請 | 申請トップ画面・ステータス表示 |
| COM-004 | 公的証明書・本人顔写真送付 | 本人確認書類のアップロード |
| COM-005 | CCUS技術者ID・本人確認番号入力 | CCUS書類アップロード + 技能者ID入力 |
| COM-006 | 退会手続き | 退会処理 |

## 対象ロール

- 受注者（Contractor）: 全機能利用可能
- 発注者（Client）: 全機能利用可能（退会は管理責任者のみ）
- 担当者（Staff、法人 Admin 含む `users.role = 'staff'`）: **COM-001 / COM-002 には直接アクセスできない**。`/profile` および `/profile/edit` は Middleware で CLI-024 自己編集モード（`/mypage/members/{自分ID}/edit`）にリダイレクトされる。理由: CLI-025 招待経由で作成された Admin / Staff は受注者登録フローを通っていないためスキル・資格等の受注者向けデータを持たず、COM-001/002 のフォームが整合しない。自己編集は organization 機能の CLI-024 に一元化（詳細は `.kiro/specs/organization/requirements.md` REQ-ORG-011 参照）。COM-003〜005（本人確認/CCUS）は受注者向け機能のため Staff のマイページから導線を表示せず、業務上の利用想定もなし。COM-006（退会）は `org_role != 'owner'` を Server Action で拒否

## 機能要件

### REQ-PF-001: プロフィール詳細（COM-001）

- 自分のプロフィール情報を表示する
- **Staff（`users.role = 'staff'`）は本画面にアクセスできない**（Middleware で CLI-024 にリダイレクト。詳細は「対象ロール」セクション）
- 表示項目（CLI-020 と同じ視覚パターン: plain div + 紫ラベルの情報テーブル + 白値セル + `min-h-[40px]`）:
  - アバター画像（`users.avatar_url` or フォールバック icon）
  - 氏名（姓 + 名）+ 年齢（`birth_date` がある場合）
  - バッジ: 本人確認済み / CCUS登録済み（承認/申請中/未承認 の 3 状態）
  - 基本情報（DetailRow 形式で列挙）: メールアドレス、会社名/屋号、お住まい、対応可能エリア、性別
  - 自己紹介（`users.bio`、空欄時はプレースホルダー表示）
  - 能力（DetailRow 形式）: 対応できる職種、経験年数、**保有スキル**、保有資格
    - 「対応できる職種」は `user_skills.trade_type`（固定リスト由来の大分類）
    - 「保有スキル」は `users.skill_tags`（自由入力の詳細タグ）— 両者は別物なので重複表示しない
  - PR動画（`users.video_url` がある場合のみ）
- 「編集する」ボタン（primary pill `w-full max-w-xs rounded-pill` centered）→ COM-002 へ遷移
- 「もどる」ボタン（BackButton `href="/mypage"`）で CON-001 へ戻る
- 「退会希望の方はこちら」リンク（小さく下部）→ COM-006 へ遷移

### REQ-PF-002: プロフィール編集（COM-002）

- プロフィール情報を編集する
- **Staff（`users.role = 'staff'`）は本画面にアクセスできない**（Middleware で CLI-024 にリダイレクト。詳細は「対象ロール」セクション）
- UI パターン: CLI-021 と同じ視覚パターンに統一（`bg-muted` ページ + 白カード、`bg-background` Input、shadcn Select、MultiSelect、FieldLabel + RequiredBadge、アバター中央配置）
- 編集可能フィールド:
  - プロフィール画像（JPEG/PNG、最大5MB）— 中央配置の円形 + 「画像を登録する」ボタン
  - 姓（必須）
  - 名（必須）
  - 性別（必須、shadcn Select）
  - 生年月日（必須、Input type=date）
  - メールアドレス（変更時は認証メール送信 → Supabase Auth の `updateUser` でメール変更）— 注意文「※ 変更時は新しいメールアドレス宛に認証用メールが送られるため認証の対応をお願いします」
  - 会社名/屋号（任意）— 注意文「※ ない場合は「なし」と入力をお願いします」
  - お住まい — 都道府県（必須、shadcn Select）
  - 対応可能エリア（必須、**MultiSelect（複数選択可能なプルダウン）**で 47 都道府県から選択）
  - 自己紹介（任意、Textarea）
  - 職種（必須、最大3つ、shadcn Select × 最大 3 行、「職種を追加」ボタンで行追加）
  - 経験年数（職種ごとに Input type=number。入力欄には列見出し「経験年数（年）」と `aria-label="経験年数 {n}（年）"` を付与し、何の数値か一目で分かるようにする）
  - **保有スキル**（任意、自由入力テキスト + chips 表示、「追加する」ボタン。`users.skill_tags text[]` に保存）
  - 保有資格（任意、自由入力テキスト + chips 表示、「追加する」ボタン。`user_qualifications` に保存）
- 更新対象テーブル: users（skill_tags 含む）, user_skills, user_qualifications, user_available_areas
- プロフィール画像は Supabase Storage（avatars バケット）にUUIDファイル名で保存
- **プロフィール画像の更新時**: 新しい画像のアップロード成功後、旧画像ファイルを Supabase Storage から削除する（ストレージ容量の節約）。削除は Server Action 内で新画像保存 → 旧画像削除の順で実行し、旧画像の削除失敗はエラーにしない（ログ記録のみ）
- 保存成功時: COM-001（詳細）へ遷移
- 「もどる」ボタン（BackButton、`href` なし → `router.back()` で COM-001 へ自然遷移）

##### 法人プラン Owner 向け注意バナー（既存仕様 R1）

`users.role = 'client'` かつ `org_role = 'owner'` のユーザーが本画面を開いたとき、画面上部に「氏名・メールアドレスの変更は同一人物の情報更新のみです。契約者（管理責任者）を別の方に引き継ぐ場合は、お問い合わせからご依頼ください」の注意バナーを表示する。編集機能自体は一切制限しない（organization spec Task 13.5 準拠）。

### REQ-PF-003: 本人確認・CCUS登録申請トップ（COM-003）

- 本人確認とCCUS登録の申請状況を表示する
- 表示項目:
  - 本人確認ステータスバッジ: 未申請 / 申請中 / 承認済み / 否認（再提出可）
  - CCUS登録ステータスバッジ: 未申請 / 申請中 / 承認済み / 否認（再提出可）
  - **否認時の追加表示:** 否認理由（rejection_reason）をアラートボックスで表示し、「再提出する」ボタンを表示する
- 「本人確認書類を提出する」ボタン → COM-004 へ遷移（未申請 or 否認時のみ活性）
- 「CCUS登録申請する」ボタン → COM-005 へ遷移（本人確認が承認済みの場合のみ活性）

**再提出フロー:**
- 否認（rejected）された場合、ユーザーは「再提出する」ボタンから COM-004 / COM-005 へ遷移できる
- 再提出時の処理は REQ-PF-004 / REQ-PF-005 と同じフォームを使用する（新規申請と同じ画面）
- 再提出回数に上限は設けない（ただし毎回 audit_logs に記録する）

### REQ-PF-004: 公的証明書・本人顔写真送付（COM-004）

- 本人確認書類をアップロードする
- 入力フィールド:
  - 書類1（必須、JPEG/PNG/PDF、最大10MB）— 公的証明書
  - 書類2（必須、JPEG/PNG/PDF、最大10MB）— 本人顔写真
- identity_verifications テーブルにデータを保存:
  - document_type = 'identity'
  - status = 'pending'
  - document_url_1, document_url_2 に Storage URL
- ファイルは Supabase Storage（identity-documents バケット）にUUIDファイル名で保存
- アップロード成功時: COM-003 へ遷移（ステータス: 申請中に更新）
- 監査ログに記録

**再提出時のデータ処理:**
- 再提出は **新しいレコードを INSERT** する（既存の rejected レコードは履歴として残す）
- UNIQUE (user_id, document_type) WHERE status = 'pending' 制約により、審査中の申請が1件のみであることが保証される
- 過去の rejected レコードは管理者が審査履歴として参照できる
- 古い書類ファイル（Storage 上のファイル）は rejected レコードの document_url から参照可能なため、削除しない（監査証跡として保持）

### REQ-PF-005: CCUS技術者ID・本人確認番号入力（COM-005）

- CCUS関連書類をアップロードし、技能者IDを入力する
- 前提条件: 本人確認が承認済みであること
- 入力フィールド:
  - CCUS書類（必須、JPEG/PNG/PDF、最大10MB）
  - CCUS技能者ID（必須、テキスト）
- identity_verifications テーブルにデータを保存:
  - document_type = 'ccus'
  - status = 'pending'
  - ccus_worker_id を設定
- ファイルは Supabase Storage（ccus-documents バケット）に保存
- アップロード成功時: COM-003 へ遷移

### REQ-PF-006: 退会手続き（COM-006）

- アカウントの退会処理を行う
- 入力フィールド:
  - 退会理由（必須、プルダウン、OptionSets から選択）
  - 詳細・改善事項（任意、テキスト）
  - 同意チェックボックス（必須）
- 退会不可条件:
  - 退会申請者本人が「応募中」（status = 'applied'）の案件がある場合（受注者立場として）
  - 退会申請者が**責任を持つ**「発注中」（status = 'accepted' で未完了）の案件がある場合
    - 個人発注者プラン Owner / 小規模プラン Owner: `jobs.owner_id = user.id` のみ判定
    - **法人プラン Owner**: `jobs.organization_id = 所属組織の id` で判定
      （**組織メンバー=Admin/Staff が作成した案件も含む**。Owner は組織全体の発注責任を負うため、自分以外が作成した組織案件の進行中を見落とすと、退会後に発注責任者不在になる）
  - 法人プラン Staff / Admin: そもそも退会不可（管理責任者に依頼。Server Action の事前ガード + Middleware でブロック）
- **法人プラン管理責任者（Owner）向けの警告ダイアログ**（2026-04-19 追加、C 案採用に伴う）:
  - 対象: `role = 'client'` かつ `org_role = 'owner'` かつ `subscriptions.plan_type IN ('corporate', 'corporate_premium')` のユーザー
  - 表示タイミング: 退会理由入力後、「退会する」ボタン押下時の確認ダイアログ内に表示
  - 文言（ユーザー向けの平易な表現。「Admin」「Staff」「CLI-026」等の内部用語は使わない）:
    > 退会すると、会社アカウント「{display_name}」は削除され、**あなたが招待した管理者・担当者のアカウントもまとめて利用停止**になります。招待された方々はビジ友にログインできなくなります。
    >
    > 一時的に料金だけ止めたい場合は、退会ではなく**「プランの解約」**をおすすめします。プランを解約すれば、後日あらためて法人プランにご契約いただくだけで、管理者・担当者のアカウント、作成したスカウト文例、受注者との過去メッセージ、すべてを元どおりに復活できます。
    >
    > 本当に退会した場合、同じ会社でビジ友を再開するには、新しく会社アカウントを作り直して、管理者・担当者をあらためて招待する必要があります（以前のスカウト文例・メッセージ履歴は引き継げません）。
    >
    > それでも退会しますか？
  - ダイアログに「プランを解約する（CLI-026 へ）」リンクと「退会する」ボタンを並べる
  - 文言の変数 `{display_name}` には `client_profiles.display_name`（社名）を差し込む
- 退会フロー:
  1. 「退会する」ボタン → 確認ポップアップ（法人プラン Owner には上記警告ダイアログを表示）
  2. 確認OK → users.deleted_at にタイムスタンプを設定（ソフトデリート）
  3. **法人プラン Owner の場合（2026-04-19 C 案採用）**: Admin の有無に関わらず、以下を連動実行
     - 所属メンバー全員（Admin / Staff）の `users.deleted_at` をセット（ログイン不可化）
     - 所属メンバー全員の `organization_members` を物理削除
     - `organizations.deleted_at` をセット（組織ソフトデリート）
     - `client_profiles` / `scout_templates` は履歴として保持（削除しない）
     - 詳細は `.kiro/specs/organization/requirements.md` の「退会（COM-006 = `/profile/withdrawal`）」セクション参照
  4. 法人プラン Owner 以外の場合: 自身の `organization_members` のみ物理削除（所属組織は残す）
  5. Supabase Auth のアカウントを無効化
  6. 課金中の場合: Stripe サブスクリプションをキャンセル
  7. 退会完了メール送信（Resend）— メール送信失敗時は本体処理（退会処理）をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）
  8. セッションを無効化し、AUTH-001 へリダイレクト

## 非機能要件

### 静的・サポートページ

#### REQ-PF-007: よくある質問（COM-007）

- カテゴリ別のQ&A形式で表示する
- アコーディオン UI（質問クリックで回答を展開）
- カテゴリ例: アカウント、案件、課金、本人確認 等
- ログイン前（AUTH-001）とログイン後（CON-001）の両方からアクセス可能

#### REQ-PF-008: お問い合わせ（COM-008）

- お問い合わせフォームを提供する
- 入力フィールド:
  - 姓（必須）、名（必須）
  - メールアドレス（必須）
  - お問い合わせ項目（必須、チェックボックス複数選択、OptionSets）
  - 内容（必須、テキストエリア）
- contacts テーブルにデータを保存
- 送信成功時: 完了メッセージを表示

#### REQ-PF-009: 利用規約（COM-009）

- 静的ページとして利用規約を表示する

#### REQ-PF-010: プライバシーポリシー（COM-010）

- 静的ページとしてプライバシーポリシーを表示する

#### REQ-PF-011: 特定商取引法に基づく表示（COM-011）

- 静的ページとして特定商取引法に基づく表示を掲載する

※ COM-007〜011 はログイン前・ログイン後の両方からアクセス可能

## 非機能要件

### セキュリティ

- RLS: 自分のプロフィールのみ編集可能。他ユーザーのプロフィールは公開情報のみ閲覧可能
- 本人確認書類: 非公開バケット（本人 + 管理者のみアクセス可）
- 書類へのアクセスは監査ログに記録
- ファイルアップロード: MIMEタイプとファイル拡張子の両方を検証
- メールアドレス変更: Supabase Auth の標準メール認証フローを使用
- PII保護: メールアドレスは他ユーザー向けに部分マスク表示

### 通知

- 本人確認 承認通知: 管理者承認時にメール送信
- 本人確認 否認通知（再提出依頼）: 管理者否認時にメール送信
- CCUS登録 承認/否認通知: 同上
- 退会完了通知: 退会処理完了時にメール送信
- メール送信失敗時: 本体処理をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）
- ステータスバッジ: マイページ（CON-001）の本人確認エリアに表示（次回アクセス時に反映）

```
CON-001 → COM-001（プロフィール詳細）→ COM-002（プロフィール編集）

CON-001 → COM-003（本人確認トップ）→ COM-004（本人確認書類送付）
                                   → COM-005（CCUS登録申請）
```

## 関連テーブル

- users: ユーザー基本情報（CRUD）— `skill_tags text[]` カラムに保有スキル（自由入力タグ）を保存

- user_qualifications: 資格（CRUD）
- user_available_areas: 対応可能エリア（CRUD）
- identity_verifications: 本人確認・CCUS申請（INSERT, SELECT）
- audit_logs: 書類アクセスログ
