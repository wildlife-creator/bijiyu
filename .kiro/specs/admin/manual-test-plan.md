# 管理者機能（ADM-001〜024）手動テスト手順書

この手順書は、admin 機能を**1画面ずつ・複数セッションに分けて**手動で確認するためのガイドです。
自動テスト（Vitest / pgTAP / Playwright）は正常系とロジックを押さえています。ここでは
**見た目・操作感・実際のデータ表示**を、あなたの目で確かめることが目的です。

---

## このガイドの使い方

- 各ステップ先頭の `- [ ]` を、確認できたら `- [x]` に変えていってください（このファイルを直接編集してOK）。
- 「気になった点」は各セッションの末尾のメモ欄に書き留めてください。次セッションで私が修正します。
- **セッションは長くなりすぎる前に切り替えて大丈夫です。** 1セッション = 1ブロック（A〜G）を目安にしています。
- 新しいセッションを始めるときは、私（Claude）に「manual-test-plan.md の続き（ブロックX）をやりたい」と伝えてください。私がこのファイルの進捗を見て、環境を整えてからご案内します。

---

## セッション開始時の準備（毎回・私が代行します）

新セッションで手動テストを始めるとき、私が次を実行して環境をクリーンにします。あなたは確認するだけでOKです。

1. Supabase が起動しているか（`supabase status`）
2. 開発サーバーが起動しているか（`npm run dev` → http://127.0.0.1:3000）
3. **`supabase db reset`** でデータを初期状態に戻す（前回の操作跡をリセット）

> 💡 **なぜ毎回リセットするの？**
> 「承認」「削除」「発注取消」などは一度実行するとデータが変わります。
> セッションの最初に必ずリセットすれば、毎回まっさらな状態から確認できます。

## 使い捨てアカウント早見表（壊してOKなアカウント）

「戻せない操作」はこのアカウントで試してください。リセットすれば元に戻ります。

| 操作 | 対象アカウント / データ | 表示名 |
|---|---|---|
| 本人確認の**承認** | `adm-pending-identity@test.local` | 山本健（本人確認 pending） |
| 本人確認の**否認** | `adm-reject-identity@test.local` | 森大地（本人確認 pending #2） |
| CCUS の承認/否認 | `adm-pending-ccus@test.local` | 井上翔（CCUS pending） |
| 発注者の**単純削除** | `adm-del-individual@test.local` | 削除テスト個人（個人発注者・案件なし） |
| 発注者の**カスケード削除** | `adm-del-corp-owner@test.local` | 削除テスト法人（法人 Owner＋担当者1名） |
| 応募の**発注取消** | 応募ID `ada00000-0000-4000-8000-000000000004` | 井上翔の応募（山田建設の発注済み案件・初回稼働日前） |
| 招待フロー | 任意の新規メール（例: `try-invite@test.local`） | 実際にメールが Mailpit に届きます |

> 💡 発注取消の対象応募は、セッション開始時の `supabase db reset` で「発注済み（accepted）」状態に戻ります。リセット前は前回の確認で取消済みになっていることがあります。

> ⚠️ **本番フィクスチャ（鈴木工務店・山田建設・中村リフォーム）は削除しないでください。**
> 他の確認項目が一緒に消えてしまいます。削除を試すときは必ず上の「削除テスト」アカウントで。
