# 管理者機能（admin）— 要件定義

## 概要

サービス運営者向けの管理機能。発注者アカウント管理、ユーザーアカウント管理、
本人確認承認、応募履歴管理、パスワード変更を提供する。
一般ユーザーとは完全に分離された認証・画面体系。

## 対象ロール

- 管理者（Admin）のみ。一般ユーザーはアクセス不可

### システム管理者の作成ルール

- **初期作成**: サービス開始前に、Supabase の管理画面（SQL Editor）からデータベースに直接登録する。手順は以下のとおり:
  1. Supabase Auth でメールアドレス・パスワードを登録（auth.users に追加）
  2. public.users の role を 'admin' に設定
- **追加作成**: 新しいシステム管理者が必要な場合も、同じ手順でデータベースに直接追加する
- **管理画面からの管理者作成機能は設けない**: セキュリティ上、管理画面のハッキングによる権限昇格（= 一般ユーザーが管理者になってしまうこと）を防ぐため
- **管理者の削除**: データベースから直接削除する（管理画面からは不可）
- **管理者同士は対等で、管理画面からお互いを削除・編集できない**: 管理画面で操作できるのは一般ユーザー（受注者・発注者）のアカウントのみ。管理者アカウントの追加・変更・削除はすべて Supabase SQL Editor で直接行う運用とする

## 機能要件

### 認証・トップ

#### REQ-ADM-001: 管理者ログイン（ADM-001）

- 管理者専用のログイン画面
- メールアドレスとパスワードでログイン
- 一般ユーザーのログイン画面（AUTH-002）とは完全に別画面・別URL（`/admin/login`）
- ログイン成功時: ADM-002 へ遷移
- 管理者以外（role != 'admin'）がログインした場合: エラーメッセージ表示
- ログイン失敗時のエラーメッセージは一般ユーザーと同様（アカウント存在の推測を防止）

#### REQ-ADM-002: 管理者トップページ（ADM-002）

- 管理機能への導線を表示する
- メニュー項目:
  - 発注者アカウント一覧 → ADM-003
  - ユーザーアカウント一覧 → ADM-008
  - 本人確認承認申請一覧 → ADM-011
  - 応募履歴一覧 → ADM-013
  - パスワード変更 → ADM-015
  - ログアウト → ADM-001
- ダッシュボード情報（オプション）:
  - 未処理の本人確認申請数
  - 本日の新規登録ユーザー数
  - 本日の新規応募数

### 発注者アカウント管理

#### REQ-ADM-003: 発注者アカウント一覧（ADM-003）

- 発注者アカウントの一覧を表示する
- 検索: キーワード検索（氏名、メールアドレス、発注者名）
- フィルター:
  - 権限（管理責任者 / 管理者 / 担当者）
  - オプションプラン加入者（急募 / 補償¥5,000 / 補償¥9,800 の単一選択）
- ページネーション: 20件ずつ
- 「管理責任者 新規登録」ボタン → ADM-006 へ遷移
- 各行クリック → ADM-004 へ遷移

#### REQ-ADM-004: 発注者アカウント詳細（ADM-004）

- 発注者の詳細情報を表示する
- 表示項目:
  - 内部管理者メモ
  - 急募オプション状態（管理責任者のみ表示）
  - 発注者情報: アイコン、会社名、住所、基本情報
  - 担当者情報: 所属する担当者一覧
- 「編集する」ボタン → ADM-005 へ遷移
- 「アカウントを削除する」ボタン:
  - 確認ポップアップ表示
  - 管理責任者を削除すると配下の担当者もすべて削除される旨を警告
  - ソフトデリート（deleted_at にタイムスタンプ設定）
  - Stripe サブスクリプションのキャンセル

#### REQ-ADM-005: 発注者アカウント編集（ADM-005）

- 管理者が発注者の情報を編集する
- 編集可能フィールド:
  - 管理者のメモ（テキストエリア）→ client_profiles.admin_memo
  - 急募オプション（チェックボックス、管理責任者のみ表示）→ client_profiles.is_urgent_option
- 保存成功時: ADM-004 へ遷移

#### REQ-ADM-006: 管理責任者 新規作成（ADM-006）

- 新規の発注者（管理責任者）を管理者が作成する
- 入力フィールド:
  - 発注者名（必須）
  - 担当者名（必須）
  - メールアドレス（必須）
- 「入力内容を確認する」ボタン → ADM-007 へ遷移

#### REQ-ADM-007: 管理責任者 新規作成確認（ADM-007）

- 作成内容を確認して実行する
- 入力内容の確認表示
- 「作成する」ボタン — Server Action 内で以下を実行:
  1. メールアドレスの重複チェック（auth.users に同じメールが既に存在しないか事前確認）
  2. `supabase.auth.admin.createUser()` で auth.users にアカウント作成（service_role キーを使用）
  3. auth.users 作成時のトリガーにより public.users が自動生成される
  4. 以下をトランザクションで実行（途中で失敗したら全部取り消す）:
     a. public.users.role を 'client' に更新
     b. client_profiles テーブルに UPSERT（存在しなければ INSERT、存在すれば display_name を更新）。display_name に入力された発注者名を設定。※ client_profiles の正規作成ルートは billing Webhook（checkout.session.completed）だが、admin 経由の場合は課金前に組織を先に作成するため、ここで UPSERT する
     c. organizations テーブルに組織作成（name に入力された発注者名を設定）
     d. organization_members テーブルに管理責任者として追加（org_role = 'owner'）
  5. トランザクション失敗時のクリーンアップ: ステップ 2 が成功済みの場合は `supabase.auth.admin.deleteUser()` で auth.users からも削除する（幽霊アカウント防止）
  6. 会員登録メール送信（パスワード設定用リンク）— メール送信失敗時は本体処理をロールバックしない
- エラー時のユーザー表示:
  - メールアドレス重複: 「このメールアドレスは既に登録されています」
  - その他のエラー: 「アカウントの作成に失敗しました。時間をおいて再度お試しください」
- 作成成功時: ADM-003 へ遷移

### ユーザーアカウント管理

#### REQ-ADM-008: ユーザーアカウント一覧（ADM-008）

- ユーザー（受注者）アカウントの一覧を表示する
- 検索: キーワード検索（氏名、メールアドレス）
- フィルター: オプションプラン加入者（動画掲載のみ）
- 各行の表示: 氏名、年齢、メールアドレス、退会済み表示
- ページネーション: 20件ずつ
- 各行クリック → ADM-009 へ遷移

#### REQ-ADM-009: ユーザーアカウント詳細（ADM-009）

- ユーザーの詳細情報を表示する
- 表示項目:
  - 「動画を投稿する」ボタン（動画掲載オプション加入者のみ表示）→ ADM-010 へ遷移
  - アイコン、氏名、年齢
  - バッジ: 本人確認済み、CCUS登録済み
  - PR動画（登録済みの場合）
  - 基本情報、自己紹介、能力
  - 発注者からの評価（user_reviews 6項目の集計）
  - 評価の補足コメント一覧（20件ずつページネーション）
- 「アカウントを削除する」ボタン:
  - 確認ポップアップ表示
  - ソフトデリート
  - 監査ログに記録

#### REQ-ADM-010: ユーザー動画投稿（ADM-010）

- 管理者がユーザーの代わりにTikTok動画URLを登録する
- 用途: ビジ友公式TikTokアカウントでユーザー紹介動画を作成し、そのURLを登録する（オプション「動画掲載」100,000円/動画）
- 前提条件: 対象ユーザーが動画掲載オプションに加入していること
- 遷移元: ADM-009（ユーザーアカウント詳細）の「動画を投稿する」ボタン
  - 動画掲載オプション未加入のユーザーでは「動画を投稿する」ボタンを非表示
- 入力フィールド:
  - TikTok動画URL（必須）— TikTokの埋め込みURLを想定
- 表示:
  - 既登録時: 「更新」ボタン + 現在の登録URL表示 + プレビュー
  - 未登録時: 「登録」ボタン
- users.video_url を更新
- 登録された動画は以下で表示:
  - COM-001（ユーザープロフィール詳細）のPR動画エリア
  - ADM-009（ユーザーアカウント詳細）のPR動画エリア
  - CLI-006（ユーザー詳細 / 発注者が見る職人プロフィール）のPR動画エリア

### 本人確認承認

#### REQ-ADM-011: 本人確認承認申請一覧（ADM-011）

- 本人確認・CCUS登録の承認待ち申請を一覧表示する
- 表示: 「全○○件」、各行: 氏名、年齢、メールアドレス
- ステータス: pending のみ表示
- ページネーション: 20件ずつ
- 各行クリック → ADM-012 へ遷移

#### REQ-ADM-012: 本人確認承認可否（ADM-012）

- 本人確認書類の承認/否認を行う
- **対象レコードの特定方法**: ADM-011 の一覧画面で選択した identity_verifications.id をパラメータとして受け取る。一覧画面では `status = 'pending'` のレコードのみ表示しているため、必ず審査待ちのレコードが対象になる。同一ユーザーが複数回申請した場合は、rejected レコードは履歴として残り、pending のレコードのみが一覧に表示される（database-schema.md の UNIQUE 制約 `(user_id, document_type) WHERE status = 'pending'` により同時に pending が2件以上になることはない）
- 表示項目: アイコン、氏名、年齢
- **本人確認セクション**:
  - 本人確認画像2枚を表示（identity-documents バケットから取得）
  - 否認理由テキスト入力フィールド
  - 否認ボタン（否認理由入力時のみ活性化）
  - 承認ボタン（常に活性化）
- **CCUS登録セクション**:
  - CCUS画像を表示（ccus-documents バケットから取得）
  - 技能者ID表示
  - 否認理由テキスト入力フィールド
  - 承認ボタン（本人確認が承認済みの場合のみ活性化）
  - 否認ボタン（本人確認承認済み + 否認理由入力時のみ活性化）
- 承認時:
  - identity_verifications.status を 'approved' に更新
  - identity_verifications.reviewed_by, reviewed_at を設定
  - users.identity_verified（または ccus_verified）を true に更新
  - ユーザーへ承認通知メール送信 — メール送信失敗時は本体処理（identity_verifications と users の更新）をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）
- 否認時:
  - identity_verifications.status を 'rejected' に更新
  - identity_verifications.rejection_reason を設定
  - ユーザーへ否認通知メール送信（再提出依頼）— メール送信失敗時は本体処理をロールバックしない（同上）
- 書類へのアクセスは監査ログに記録:
  - **記録タイミング**: ADM-012 画面を開いた際、書類の表示用URL（署名付きURL = 有効期限つきのアクセス許可リンク）を生成する Server Action の中で audit_logs テーブルに INSERT する
  - **記録内容**: action = 'identity_access'、actor_id = 操作した管理者のID、target_type = 'identity_verifications'、target_id = 対象の申請レコードID、metadata に document_type（'identity' / 'ccus'）を含める
  - **署名付きURLの有効期限**: 1時間

### 応募履歴管理

#### REQ-ADM-013: 応募履歴一覧（ADM-013）

- 全応募の一覧を表示する
- 「CSV出力」ボタン（出力項目は未定）
- 検索: キーワード検索
- フィルター: ステータス絞り込み
- ソート機能
- ページネーション: 20件ずつ
- 各行クリック → ADM-014 へ遷移

#### REQ-ADM-014: 応募履歴詳細（ADM-014）

- 応募の詳細情報を表示する
- 表示項目:
  - ステータスバッジ
  - 「発注取り消し」ボタン（条件: ステータスが 'accepted' かつ初回稼働日前の場合のみ）
    → applications.status を 'cancelled' に更新
  - 案件情報（クリックで CLI-002 へ遷移）
  - ユーザー情報（クリックで ADM-009 へ遷移）
  - 初回勤務日
  - ユーザー評価（user_reviews 6項目）
  - 発注者評価（client_reviews 1項目）

### パスワード変更

#### REQ-ADM-015: 管理者パスワード変更（ADM-015）

- 管理者自身のパスワードを変更する
- 入力フィールド:
  - 現在のパスワード（必須）
  - 新しいパスワード（必須、8文字以上）
  - 新しいパスワード確認（必須、一致確認）
- 現在のパスワードの照合後、Supabase Auth の `updateUser` で更新
- 変更成功時: 完了メッセージを表示

## 非機能要件

### セキュリティ

- Middleware: `/admin/*` ルートは role = 'admin' のみアクセス可能
- 一般ユーザーのトークンでは管理者画面にアクセス不可
- 本人確認書類へのアクセスは監査ログに必ず記録
- アカウント削除操作は監査ログに記録
- 管理者の全操作を監査ログに記録

### データ保護

- 本人確認書類は非公開バケットに保存（管理者 + 本人のみアクセス可）
- 表示時は署名付きURLを生成（有効期限付き）
