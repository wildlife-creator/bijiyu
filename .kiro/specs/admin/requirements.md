# 管理者機能（admin）— 要件定義

## 概要

サービス運営者向けの管理機能。発注者アカウント管理、ユーザーアカウント管理、
本人確認承認、応募履歴管理、ユーザーからの問い合わせ閲覧（お問い合わせ・トラブル報告・
求人へのお問い合わせ）、パスワード変更を提供する。
一般ユーザーとは完全に分離された認証・画面体系。

## 対象ロール

- 管理者（Admin）のみ。一般ユーザーはアクセス不可

### システム管理者の作成ルール

- **初期作成**: サービス開始前に、Supabase の管理画面（SQL Editor）からデータベースに直接登録する。手順は以下のとおり:
  1. Supabase Auth でメールアドレス・パスワードを登録（auth.users に追加）
  2. public.users の role を 'admin' に設定
- **追加作成**: 新しいシステム管理者が必要な場合も、同じ手順でデータベースに直接追加する
- **管理画面からの管理者作成機能は設けない**: セキュリティ上、管理画面のハッキングによる権限昇格（= 一般ユーザーが管理者になってしまうこと）を防ぐため
- **管理者の削除**: データベースから直接削除する（管理画面からは不可）
- **管理者同士は対等で、管理画面からお互いを削除・編集できない**: 管理画面で操作できるのは一般ユーザー（受注者・発注者）のアカウントのみ。管理者アカウントの追加・変更・削除はすべて Supabase SQL Editor で直接行う運用とする

## 機能要件

### 認証・トップ

#### REQ-ADM-001: 管理者ログイン（ADM-001）

- 管理者専用のログイン画面
- メールアドレスとパスワードでログイン
- 一般ユーザーのログイン画面（AUTH-002）とは完全に別画面・別URL（`/admin/login`）
- ログイン成功時: ADM-002 へ遷移
- 管理者以外（role != 'admin'）がログインした場合: エラーメッセージ表示
- ログイン失敗時のエラーメッセージは一般ユーザーと同様（アカウント存在の推測を防止）
- **「パスワードを忘れた方はこちら」リンクを設置する**（カンプ通り。2026-06-11 決定・案A）: 一般ユーザーのパスワード再設定フロー（リセットメール → 再設定画面）を流用する。再設定完了後は `/admin/login` からログインし直す
- **実装現状とのギャップ（本 spec で解消）**: 現状は専用ログイン画面が無く、通常の `/login` でログイン → middleware が `role='admin'` を検出して `/admin/dashboard` へリダイレクトする実装になっている。本 spec で `/admin/login` の専用画面を新規実装し、middleware の振り分け・seed を調整する（詳細は「実装現状とのギャップ」セクション参照）

#### REQ-ADM-002: 管理者トップページ（ADM-002）

- 管理機能への導線を表示する
- メニュー項目:
  - 発注者アカウント一覧 → ADM-003
  - ユーザーアカウント一覧 → ADM-008
  - 本人確認承認申請一覧 → ADM-011
  - 応募履歴一覧 → ADM-013
  - お問い合わせ一覧 → ADM-016
  - トラブル報告一覧 → ADM-018
  - 求人問い合わせ一覧 → ADM-020
  - メッセージ一覧（代理メッセージの閲覧）→ ADM-023（全社・絞り込みなしで開く。**最初からメニューに置く**。2026-06-11 決定・旧「入口②任意」を確定化）
  - パスワード変更 → ADM-015
  - ログアウト → ADM-001
- **ログアウト導線を必ず設置する**: 現状 admin シェルには共通ヘッダーが無くログアウトボタンが欠落しており、Cookie 削除でしか抜けられない。本 spec で全 admin 画面から到達できるログアウト導線（`logoutAction` 呼び出し）を設置する
- **件数表示・ダッシュボード数値は付けない**（2026-06-11 決定・案B＝カンプ通りメニューのみ。旧記述でオプション扱いだった「未処理の本人確認申請数・本日の新規登録ユーザー数・本日の新規応募数」の表示は不採用）

### 発注者アカウント管理

#### REQ-ADM-003: 発注者アカウント一覧（ADM-003）

- 発注者側のアカウントを**人単位で1行ずつ**表示する（2026-06-11 決定・案②）
- **対象**:
  - `role = 'client'`（個人発注者・小規模・法人の管理責任者）
  - `role = 'staff'`（法人の組織管理者・担当者）← **スタッフも1行として表示する**
  - 除外: 受注者のみの人（`role='contractor'`）＝ADM-008 側に出す ／ 運営（`role='admin'`）
  - 退会済み（`deleted_at`）は「退会済み」表示で出す
- **各行の表示**:
  - 氏名（本人の姓名）
  - 会社名（所属発注者の `display_name`。**スタッフ行は所属会社名**＝所属組織 Owner の display_name を解決して表示）
  - メールアドレス
  - 区分（下記5区分）
  - プラン（個人 / 小規模 / 法人 / 法人・高サポート。※法人と高サポートはここで見分ける）
  - **オプション加入バッジ**（active な 急募 / 職場紹介動画 をバッジ表示。**急募オプション加入が一目で分かるようにする**。2026-06-11 追加。補償は受注者向けのため含めない）
  - 退会済み表示（該当時のみ）
- **検索**: キーワード検索（氏名、メールアドレス、会社名）
- **フィルター（2枠・各単一選択）**:
  - **枠1 区分（権限・種別）**:
    - 管理責任者（`role=client` ＋ 法人/高サポートプラン＝`org_role=owner`）
    - 組織管理者（`role=staff` ＋ `org_role=admin`）
    - 担当者（`role=staff` ＋ `org_role=staff`）
    - 個人発注者（`role=client` ＋ individual プラン）
    - 小規模発注者（`role=client` ＋ small プラン）
    - ※ `org_role='admin'` は運営のシステム管理者と区別するため**画面では「組織管理者」と表記**する（2026-06-10 決定）
  - **枠2 オプションプラン加入者**:
    - 急募オプション（`option_type='urgent'`, active）
    - 動画掲載＝職場紹介（`video_workplace`, active）
    - ※ 補償¥5,000 / ¥9,800（compensation）は**受注者（職人）の給与未払い保険**で発注者活動とは無関係のため、発注者一覧には含めない（受注者一覧 ADM-008 のフィルタに置く）。2026-06-11 決定
    - ※ 受注者PR動画（`video`）も受注者側のため本画面には含めない（ADM-008 側）
- ページネーション: 20件ずつ
- 「管理責任者 新規登録」ボタン → ADM-006 へ遷移
- 各行クリック → ADM-004 へ遷移（**ADM-004 は会社＝契約主体単位の1ページ**。2026-06-11 決定。管理責任者・個人発注者・小規模発注者の行は本人＝契約主体のページへ、組織管理者・担当者の行は**所属組織 Owner の契約主体ページ**へ解決して遷移する）

#### REQ-ADM-004: 発注者アカウント詳細（ADM-004）

- **会社＝契約主体単位の1ページ**（2026-06-11 確定）。デザインカンプ `design-assets/screens/ADM-004.png` を土台に、下記の追加要素を載せる
- **ヘッダーは現状の admin 共通レイアウト**（`src/app/admin/layout.tsx` の「ビジ友 管理画面」バー＋本 spec で追加するログアウト導線）を使う。カンプの LOGO / ハンバーガー / ＜ は使わない
- **表示構成（上から順）**:
  1. 編集するボタン → ADM-005
  2. 内部管理者のメモ（`client_profiles.admin_memo`。編集は ADM-005）
  3. オプション加入状況: 急募（☑＋有効期限）／職場紹介動画の加入有無（※補償は受注者向けのため発注者詳細には出さない）
  4. 発注者情報: アイコン（image_url）・社名（display_name）・住所（address）・**プラン**（個人 / 小規模 / 法人 / 法人・高サポート）
  5. **職場紹介動画**: active な `video_workplace` の場合のみ、発注者情報の下・基本情報の上に動画を埋め込み表示（CON-006 と同じ配置）＋「職場紹介動画を投稿/編集する」ボタン → **ADM-010B**
  6. 基本情報: 募集職種・募集エリア・従業員規模・求める働き方・言語・利用SNS（client_profiles 由来）
  7. メッセージ: `client_profiles.message`（発注者が CLI-021 で入力する紹介文。**ADM-004 では閲覧専用**。運営は編集しない）
  8. 評判: 「また仕事を受けたい 👍 {good件数}」（`fetchClientReputation` の goodCount。法人は会社単位）
  9. **担当者一覧**（法人プランのみ）: 組織メンバーを複数行で表示。各行＝氏名／メールアドレス／区分（管理責任者・**組織管理者**・担当者）／招待中バッジ。**閲覧のみ**（個別の追加・削除・編集は会社側の CLI-022/023 が担当。admin はアカウント削除で配下ごと削除する）。個人・小規模は組織が無いため非表示
  10. **募集現場一覧＋集計**: 会社の掲載案件（案件数・応募数）を表示。各現場 → 募集現場詳細（ADM-022）→ 応募一覧（ADM-013 を会社/現場で絞り込み）→ 応募詳細（ADM-014）。詳細は後述「REQ-ADM-022／募集現場ドリルダウン（案B）」
  11. **代理メッセージを見る**（法人で代理メッセージありの場合のみ表示）→ メッセージ一覧（ADM-023）を**この会社で絞り込んだ状態**で開く
  12. アカウントを削除する: 確認ポップアップ／管理責任者を削除すると配下のスタッフもすべて削除される旨を警告／ソフトデリート（deleted_at にタイムスタンプ）／Stripe サブスクリプションのキャンセル（招待フローで発行された発注者も、決済済みなら通常の Stripe サブスクを持つ）
  13. もどる
- **会社単位の集計（org-scoping 準拠）**: 法人プランは案件・応募・評判を `organization_id` 単位、個人・小規模は `owner_id` 単位で集計する。発注者表示名は `resolveParticipantName()` / `client_profiles.display_name` で解決する
- **発注者の評判表示**: 受注者からの評価（client_reviews）は client-review-completion の集計（`fetchClientReputation`: good 件数・合計件数。法人は会社単位）を用いる
- **プランによる出し分け**: 法人 / 高サポート＝全表示（担当者一覧・代理メッセージあり）／個人・小規模＝担当者一覧・代理メッセージは非表示・集計は owner 単位

#### REQ-ADM-022: 募集現場詳細（ADM-022・admin 閲覧専用）／募集現場ドリルダウン（案B）

- ADM-004 の「募集現場一覧」から、会社の案件を運営が辿って閲覧するためのドリルダウン。**発注者向けの既存画面（CLI-001/002/007B）は流用せず、admin 専用に作る**（2026-06-11 決定・案B）
  - 理由: 発注者画面は「自分の会社だけ」を RLS で見る作りで、運営は全社横断で見る必要があり `createAdminClient` が必要＝データ取得が根本的に別。既存の有料画面に admin 分岐を足すと回帰リスクが高い
- **連鎖**:
  1. 募集現場一覧 … ADM-004 内のセクション（その会社の案件・件数・応募数。新規画面は作らない）
  2. **募集現場詳細（ADM-022）… admin 専用・閲覧専用の新規画面**（本ドリルダウンで唯一の新規画面）。案件内容を運営が閲覧する。発注者操作（編集・発注等）は持たない。`createAdminClient` で会社単位に取得
  3. 応募一覧 … **ADM-013（応募履歴一覧）を会社/現場で絞り込んで**流用
  4. 応募詳細 … **ADM-014（応募履歴詳細）**

#### REQ-ADM-005: 発注者アカウント編集（ADM-005）

- 管理者が発注者の情報を編集する。**編集できるのは管理者メモのみ**（2026-06-11 決定・案A）
- 編集可能フィールド:
  - 管理者のメモ（テキストエリア）→ `client_profiles.admin_memo`
- **急募オプションの編集は持たない**: 急募は「案件ごとに購入し7日間有効」の仕組みで、`client_profiles.is_urgent_option` はその表示用キャッシュ（期限切れバッチが自動更新）。手動トグルは仕組みとズレ、バッチに上書きされるため編集対象にしない。**加入状態の確認は ADM-004 のオプション加入状況（閲覧）で行う**
- 保存成功時: ADM-004 へ遷移

#### REQ-ADM-006: 管理責任者 新規作成（ADM-006）

- 営業経由などで獲得した発注者を、運営が直接アカウント発行するための画面（招待フロー）。**通常の受注者新規登録（AUTH-001〜006）をスキップ**させ、招待された本人が最短で発注者になれるようにする（目的・2026-06-10 確定）
- **作成するのはアカウント＋招待のみ**。発注者化（role=client・組織・課金レコード）はこの画面では行わない。**プラン選択と決済は招待された本人が行う**（2026-06-10 決定。案①の「運営が課金まで代行」案は不採用）
- 入力フィールド:
  - **会社名（必須）** — 決済後に `client_profiles.display_name` へ反映される。営業が把握済みの社名を運営が入力
  - **担当者氏名 姓（必須）/ 名（必須）** — `users.last_name` / `users.first_name` にセット。これが無いと middleware が「登録未完了」と判定してループするため必須
  - **メールアドレス（必須）**
- 「入力内容を確認する」ボタン → ADM-007 へ遷移
- ※ 画面名は「管理責任者」だが、招待された本人が選ぶプランによっては**個人発注者（組織なし＝管理責任者ではない単独発注者）**になりうる。これは許容する（全プラン選択可。2026-06-10 決定。名称は当面そのまま）

#### REQ-ADM-007: 管理責任者 新規作成確認（ADM-007）

- 作成内容を確認して実行する
- 入力内容の確認表示
- 「作成する」ボタン — Server Action 内で以下を実行（**2026-06-10 全面改訂**。従来の「role=client＋組織＋課金レコードを作る」手順は廃止）:
  1. メールアドレスの重複チェック（auth.users に同じメールが既に存在しないか事前確認）
  2. `supabase.auth.admin.inviteUserByEmail()`（service_role キー使用）で auth.users にアカウント作成＋招待メール送信。メタデータに `invited_role` 等を**付けない**（`handle_new_user` トリガーが `role='staff'` 化するのを避けるため）。**role は `contractor` のまま**（課金前なので発注者にしない）
  3. auth.users 作成時のトリガーにより public.users が `role='contractor'` で自動生成される
  4. `users.last_name` / `users.first_name` に入力された担当者氏名（姓・名）をセットする（middleware の「登録完了」判定〔`last_name IS NULL` を未完了扱い〕を満たし、招待後ログイン時に `/register` へ弾かれないようにするため）
  5. 入力された会社名を、**決済後に `client_profiles.display_name` へ反映できるよう一時保持**する（保持方法は design で決定。例: 招待用の保留レコード、または auth `user_metadata`。`users.company_name`〔受注者の屋号用〕は流用しない）
  6. **subscription・組織・`role=client` はここでは作らない**（本人の決済時に通常の Webhook `checkout.session.completed` が作成する）
  7. 招待メール送信失敗時のクリーンアップ: auth アカウント作成済みなら `supabase.auth.admin.deleteUser()` で削除（幽霊アカウント防止）
- エラー時のユーザー表示:
  - メールアドレス重複: 「このメールアドレスは既に登録されています」
  - その他のエラー: 「アカウントの作成に失敗しました。時間をおいて再度お試しください」
- 作成成功時: ADM-003 へ遷移

#### 招待された本人のフロー（決済以降は通常の課金フローに合流）

ADM-007 で送られた招待を受けた本人の動線。**決済から先は通常の課金フローと完全に同一**（2026-06-10 確定）。

1. 招待メールのリンク → パスワード設定（implicit flow ＋ フラグメントトークン方式。AUTH-008 招待フローと同パターン）
2. ログイン後、**受注者プロフィール入力（職種/スキル/エリア）はスキップ**し、**プラン案内（CLI-026）へ直行**する（routing を本 spec で実装。受注者オンボへ流さない）
3. 本人がプランを選択 → Stripe 決済

#### REQ-ADM-012: 本人確認承認可否（ADM-012）

- 本人確認書類の承認/否認を行う
- **対象レコードの特定方法**: ADM-011 の一覧画面で選択した identity_verifications.id をパラメータとして受け取る。一覧画面では `status = 'pending'` のレコードのみ表示しているため、必ず審査待ちのレコードが対象になる。同一ユーザーが複数回申請した場合は、rejected レコードは履歴として残り、pending のレコードのみが一覧に表示される（database-schema.md の UNIQUE 制約 `(user_id, document_type) WHERE status = 'pending'` により同時に pending が2件以上になることはない）
- 表示項目: アイコン、氏名、年齢
- **本人確認セクション**:
  - 本人確認画像2枚を表示（identity-documents バケットから取得）
  - 否認理由テキスト入力フィールド
  - 否認ボタン（否認理由入力時のみ活性化）
  - 承認ボタン（常に活性化）
- **CCUS登録セクション**:
  - CCUS画像を表示（ccus-documents バケットから取得）
  - 技能者ID表示
  - 否認理由テキスト入力フィールド
  - 承認ボタン（本人確認が承認済みの場合のみ活性化）
  - 否認ボタン（本人確認承認済み + 否認理由入力時のみ活性化）
- **審査対象でない側のセクションの表示**（2026-06-11 確定。両セクションの状態は仕組み上自動的に決まる）:
  - 本人確認の審査中 → CCUS セクションは必ず「未申請」表示（ボタン非活性＝カンプのグレーアウト）
  - CCUS の審査中 → 本人確認セクションは必ず承認済み表示（画像＋「承認済み」。ボタンは表示しない）
- 承認/否認の完了後は ADM-011（申請一覧）へ戻る
- 承認時:
  - identity_verifications.status を 'approved' に更新
  - identity_verifications.reviewed_by, reviewed_at を設定
  - users.identity_verified（または ccus_verified）を true に更新
  - ユーザーへ承認通知メール送信 — メール送信失敗時は本体処理（identity_verifications と users の更新）をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）
- 否認時:
  - identity_verifications.status を 'rejected' に更新
  - identity_verifications.rejection_reason を設定
  - ユーザーへ否認通知メール送信（再提出依頼）— メール送信失敗時は本体処理をロールバックしない（同上）
- 書類へのアクセスは監査ログに記録:
  - **記録タイミング**: ADM-012 画面を開いた際、書類の表示用URL（署名付きURL = 有効期限つきのアクセス許可リンク）を生成する Server Action の中で audit_logs テーブルに INSERT する
  - **記録内容**: action = 'identity_access'、actor_id = 操作した管理者のID、target_type = 'identity_verifications'、target_id = 対象の申請レコードID、metadata に document_type（'identity' / 'ccus'）を含める
  - **署名付きURLの有効期限**: 1時間

### 問い合わせ閲覧（運営対応）

これらの画面は support spec / job-inquiry spec で「データの器」（テーブル・RLS・Storage バケット）まで実装済み。本 spec で admin 閲覧画面を新設する。

**全画面共通の方針:**
- **閲覧のみ。対応状況（未対応/対応済）等のステータス管理・返信機能は持たない**（ユーザー決定 2026-06-09 の 1-A）。これらのテーブルに状態列を新たに追加しない
- 既存の users パターン（`createAdminClient` + サーバー側フィルタ + 20件ページング、URL searchParams を Single Source of Truth）を踏襲する
- SELECT は admin のみ（RLS 既設）。電話番号・メールアドレス等の PII は admin 画面でのみ表示する
- 添付ファイルは非公開バケット `support-attachments` から有効期限付き署名付きURLを生成して表示する（job_inquiries は添付非対応）

**表示の統一ルール（3ペア共通。2026-06-10 確定）:**
- 一覧は **受信日時の新しい順**・**20件ページング**・**絞り込みは今回なし**（将来追加検討）で統一する

- 添付の表示は **画像はインラインプレビュー / PDF はリンクで開く** で統一する（添付ありの contacts / trouble_reports 共通）
- 紐付くユーザーがいれば **ADM-009（ユーザー詳細）への導線**を詳細画面に設ける（求人問い合わせは加えて宛先発注者 → ADM-004 への導線も設ける）

## 非機能要件

### セキュリティ

- Middleware: `/admin/*` ルートは role = 'admin' のみアクセス可能
- 一般ユーザーのトークンでは管理者画面にアクセス不可
- 本人確認書類へのアクセスは監査ログに必ず記録
- アカウント削除操作は監査ログに記録
- 管理者の全操作を監査ログに記録
- お問い合わせ（contacts）・トラブル報告（trouble_reports）・求人問い合わせ（job_inquiries）の SELECT は admin のみ（RLS 既設）。電話番号・メールアドレス等の PII は admin 画面でのみ表示する

### データ保護

- 本人確認書類は非公開バケットに保存（管理者 + 本人のみアクセス可）
- 表示時は署名付きURLを生成（有効期限付き）
- 問い合わせ添付は非公開バケット `support-attachments`（admin 専用）に保存。表示は署名付きURL
