# Implementation Plan

> 本機能は既存画面の「データの絞り込み範囲」を会社単位に揃える横展開が主体で、画面の見た目は変更しない（新規デザインカンプ参照は不要）。DB に関わるのは ①評価の列追加（案C）と ②昇格処理 `ensure_organization_exists` への組織ID自動付与（その発注者の既存データに会社IDを設定・冪等・不可逆操作なし）の2つ。
> 依存関係: 読み取り/認可の各画面修正（タスク3・4・5）の**コード**は評価基盤（タスク2）のコードに依存しない。昇格ユーザーの既存データは、タスク2の昇格時付与（2.5/2.6）で会社IDが付く。CLI-020 評判（タスク6）はタスク2の集計拡張に依存する。

- [x] 1. 着手前ベースライン: 既存テスト全実行・デグレ確認
  - `npm run test`（Vitest）/ `supabase test db`（pgTAP）/ `npm run test:e2e`（Playwright）を順に実行し、全てパスすることを確認する
  - 失敗があれば原因を調査・修正してから実装に着手する
  - _Requirements: 10.4_

- [x] 2. 発注者評価の会社単位化（案C）＋ 昇格前データの組織ID補正（backfill）
- [x] 2.1 client_reviews に organization_id 列を追加
  - nullable な会社参照列を追加（組織削除時は NULL 化）し、部分インデックスを付与する（既存行の backfill は jobs 補正後に 2.6 で実施）
  - `user_reviews`（受注者への評価）には列を追加しない。既存 RLS（`can_view_client_review` 等）は変更しない
  - _Requirements: 7.1, 7.6, 7.7, 1.3_
- [x] 2.2 Supabase 型定義の再生成
  - 列追加に合わせて自動生成型を更新し、以降の挿入・集計コードが新列を型安全に扱えるようにする（書き込み・集計の実装より前に必須）
  - 前提: 2.1
  - _Requirements: 7.1_
- [x] 2.3 (P) 評価作成時に会社IDを保存
  - 受注者→発注者評価の作成処理（`submitContractorReportAction`、既に admin クライアント使用）で、対象応募の案件の会社IDを評価行に保存する（個人発注者案件は NULL）
  - 被評価者（案件作成者）は従来どおり保持し、作成者別の内訳を残す
  - 前提: 2.2（型再生成後でないと挿入の型が通らない）。2.4 とは別ファイルのため並行可
  - _Requirements: 7.2, 7.3, 7.5, 1.1_
- [x] 2.4 (P) 評判集計を組織/個人スコープに拡張
  - 評判集計関数を「組織なら会社ID軸、個人発注者なら被評価者ID軸」の判別ユニオン入力に拡張する（純粋な集計ロジックは不変）
  - 組織スコープの読み取りには admin（service-role）クライアントを渡す前提とする（既存 RLS は被評価者の現在所属で判定し、削除済み担当者の評価を弾くため。RLS は変更しない）
  - 前提: 2.2。2.3 とは別ファイルのため並行可
  - _Requirements: 8.1, 8.2, 8.3_
- [x] 2.5 昇格処理に組織ID自動付与を追加（jobs / scout_templates / job_inquiries）
  - `ensure_organization_exists` を `CREATE OR REPLACE` で拡張し、組織確定後にその発注者の「組織ID空」の jobs / scout_templates / job_inquiries に組織IDを設定する（冪等＝NULL 行のみ対象）
  - 対象列に注意: jobs / scout_templates は `owner_id = uid AND organization_id IS NULL`、job_inquiries は `target_client_id = uid AND target_organization_id IS NULL`（`target_user_id` という列は存在しない）
  - SECURITY DEFINER + `SET search_path = public` を維持する（既存定義を踏襲。CLAUDE.md ルール）
  - 個人発注者はこの関数が呼ばれないため対象外（NULL 維持）。昇格後に旧案件・旧テンプレ・旧問い合わせが会社単位に含まれ、将来の昇格でも自動で正しくなる
  - message_threads には同関数に UPDATE を**追加しない**（昇格付与の対象外＝Req 11.2。位置≠役割で「発注者側スレッドだけ」を安全選別できないため）
  - _Requirements: 11.1, 11.2, 11.4, 11.5_
> 注: message_threads（チャット）は昇格時付与の**対象外**。個人スレッドは participant_1/participant_2 の位置が役割（発注者/受注者）を表さず「発注者側スレッドだけ」を安全に選別できないため。昇格後の新規スレッドは作成時に会社IDが付与され会社共有されるため実害なし（Req 11.2・2026-06-04 決定）。
- [x] 2.6 昇格処理に client_reviews の組織ID付与を追加（jobs 付与の後）
  - 同関数内で、jobs 付与の後に「評価 → 応募 → 案件の会社」を辿って「組織ID空」の評価に組織IDを設定する（個人発注者の案件は NULL のまま）
  - 前提: 2.1（列追加）・2.5（jobs 付与）
  - _Requirements: 7.4, 11.3, 11.5_

- [x] 5. (P) CLI-014 一斉送信の宛先を会社単位に（サーバー移譲）
  - 宛先候補の収集をサーバー側に移し、閲覧者が組織所属なら自組織の全スレッドの相手（職人）を候補とする（個人発注者は従来どおり自分が当事者のスレッドのみ）
  - 組織スレッドの相手は職人（participant_2）に固定して特定し、同一の職人を重複表示しない
  - 既存の対話フォームはクライアント子コンポーネントへ分離し、候補と自ユーザーIDを受け取る。送信処理は既に会社単位対応のため変更しない（メッセージ RLS も不変）
  - 会社単位の収集は `message_threads.organization_id = org` で行う。昇格前から存在する個人スレッドは会社IDが付与されない設計（Req 11.2 で対象外）のため宛先に含まれないが、昇格後に作成される新規スレッドは作成時点で会社IDが付くため会社収集に含まれる（dev/seed は会社設定済みデータのため影響なし）
  - フォーム内ボタンの type 明示（CLAUDE.md の暗黙送信回避ルール）に留意
  - _Requirements: 6.1, 6.2, 6.3, 6.4, 6.5, 1.2_
