# Implementation Plan

- [x] 1. 認証基盤セットアップ
- [x] 1.1 Supabase クライアントヘルパーの作成
  - サーバーコンポーネント・Server Actions 用の Cookie ベースクライアントを作成する
  - クライアントコンポーネント用のブラウザクライアントを作成する
  - Middleware 専用のクライアント（レスポンスに Cookie を書き込み可能）を作成する
  - 全クライアントで Supabase の型定義（自動生成済み）を活用する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1, 8.1_
  - _Contracts: AuthMiddleware Service, LoginAction Service_

- [x] 1.2 (P) 共通レスポンス型と Zod バリデーションスキーマの定義
  - Server Actions の統一レスポンス型（成功/失敗の判別可能な型）を定義する
  - ログインフォーム用スキーマ（メール形式 + パスワード必須）を定義する
  - メール認証フォーム用スキーマ（メール形式）を定義する
  - パスワードリセット申請用スキーマ（メール形式）を定義する
  - パスワード再設定用スキーマ（8文字以上 + 確認入力一致）を定義する
  - プロフィール入力用スキーマ（姓名必須、職種1〜3件、対応エリア1件以上、パスワード8文字以上）を定義する
  - エラーメッセージはすべて日本語で記述する
  - _Requirements: 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 1.3 (P) 選択肢定数データの定義
  - 職種リスト（大工、電気工事士等）を TypeScript 定数として定義する
  - 47都道府県リストを定義する
  - 性別の選択肢を定義する
  - プロフィール入力フォームとバリデーションで共用する
  - _Requirements: 6.1_

- [x] 2. Middleware による認証・認可ルーティング制御
- [x] 2.1 セッション検証と DB ロール取得
  - 全リクエストで Supabase Auth のセッショントークンをリフレッシュし Cookie を更新する
  - 公開ルート（認証画面、Webhook エンドポイント、静的アセット）はセッション検証をスキップする
  - 認証済みユーザーについて、DB から role・削除状態・有効フラグを毎回取得する（JWT 内の role は信用しない）
  - is_active が false の場合、セッションを破棄してログイン画面にリダイレクトする（「アカウントが一時停止されています。詳しくは管理者にお問い合わせください」）
  - _Requirements: 1.1, 2.1, 8.1_
  - _Contracts: AuthMiddleware Service_

- [x] 2.2 ロール別ルーティングルールの適用
  - 未認証ユーザーは認証画面以外をすべてログイン画面にリダイレクトする
  - 認証済みユーザーが認証画面にアクセスした場合はマイページにリダイレクトする
  - 受注者は管理者画面と発注者専用画面をブロックする（課金導線 CLI-026〜027 は例外で許可）
  - 発注者・担当者は管理者画面をブロックする
  - 管理者は管理者画面のみアクセス可とし、一般ユーザー画面にはアクセス不可とする
  - ルーティングルールの一覧をパターンマッチ方式で定義し、拡張しやすい構造にする
  - _Requirements: 1.1, 2.1, 8.1_
  - _Contracts: AuthMiddleware Service_

- [x] 3. (P) Auth Callback ルートハンドラーの実装
  - メール確認・パスワードリセットのリンクを受け取る GET エンドポイントを作成する
  - クエリパラメータの認証コードを使って Supabase Auth と PKCE トークン交換を行う
  - フロー種別を判定し、新規登録の場合はプロフィール入力画面へ、パスワードリセットの場合は再設定画面へリダイレクトする
  - 認証コードが無い、またはトークン交換に失敗した場合はログイン画面にリダイレクトする
  - _Requirements: 4.1, 5.1_
  - _Contracts: AuthCallbackHandler API_

- [x] 4. ログイン機能
- [x] 4.1 (P) ログインフォーム画面の実装
  - メールアドレスとパスワードの入力フォームを作成する（Client Component）
  - Zod スキーマによるクライアント側リアルタイムバリデーションを実装する
  - 送信時にローディング状態を表示し、二重送信を防止する
  - エラーメッセージをフォーム上に日本語で表示する
  - 「パスワードをお忘れの方」リンクをパスワードリセット申請画面（AUTH-003）へ設置する
  - 「新規登録はこちら」リンクをメール認証画面（AUTH-005）へ設置する
  - _Requirements: 2.1_

- [x] 4.2 ログイン Server Action と監査ログ記録の実装
  - サーバー側で Zod バリデーションを実施し、Supabase Auth でメール/パスワード認証を行う
  - ログイン成功時に DB からユーザーの role を取得し、admin の場合は管理者トップへ、それ以外はマイページへ遷移先を返す
  - エラーメッセージは一律「メールアドレスまたはパスワードが正しくありません」を返す（アカウント存在推測の防止）
  - ログイン成功時は audit_logs に action='auth.login.success' を記録する
  - ログイン失敗時は audit_logs に action='auth.login.failure' を記録する（メールアドレスをマスクして metadata に含める。パスワードやトークンは絶対に含めない）
  - audit_logs への書き込み失敗でログイン処理自体をロールバックしない
  - _Requirements: 2.1_
  - _Contracts: LoginAction Service_

- [x] 5. 新規登録メール認証
- [x] 5.1 (P) メール認証画面の実装
  - メールアドレス入力フォームを作成する（Client Component）
  - Zod スキーマによるクライアント側バリデーションを実装する
  - 送信成功後に「認証メールを送信しました。メールのリンクをクリックしてください」メッセージを表示する
  - _Requirements: 5.1_

- [x] 5.2 メール認証 Server Action の実装
  - サーバー側で Zod バリデーションを実施する
  - Supabase Auth の signUp をサーバー側で生成した暗号学的に安全なランダム文字列を一時パスワードとして呼び出す（最低32文字。クライアントに送信しない。ログにも記録しない）
  - メール確認リンクのリダイレクト先を Auth Callback エンドポイントに設定する
  - 既に登録済みのメールアドレスでも同じ成功メッセージを返す（アカウント存在推測の防止）
  - _Requirements: 5.1_
  - _Contracts: SignupAction Service_

- [x] 6. 新規登録プロフィール入力
- [x] 6.1 (P) complete_registration データベース関数のマイグレーション作成
  - ユーザー基本情報の更新、職種データの登録（最大3件）、対応可能エリアの登録をトランザクション内で一括実行する PostgreSQL 関数を作成する
  - SECURITY DEFINER で実行する（RLS バイパスが必要。入力値の検証は Server Action で実施済みの前提）
  - マイグレーションファイルを作成し、既存のマイグレーション（001〜005）の後に配置する
  - _Requirements: 6.1_
  - _Contracts: CompleteRegistrationAction Service_

- [x] 6.2 プロフィール入力フォーム画面の実装
  - 姓・名・性別・生年月日・都道府県・会社名（任意）・職種選択・経験年数・対応可能エリア・パスワードの入力フォームを作成する（Client Component）
  - 職種は最大3つまで選択可能とし、各職種に対して経験年数を設定できるようにする
  - 対応可能エリアは都道府県の複数選択を実装する
  - 選択肢データ（職種・都道府県・性別）はタスク1.3で定義した定数を使用する
  - Zod スキーマによるクライアント側バリデーションを実装する
  - _Requirements: 6.1_

- [x] 6.3 プロフィール登録 Server Action の実装
  - セッションからユーザー ID を取得する（未認証の場合はエラーを返す）
  - サーバー側で Zod バリデーションを実施する
  - role パラメータはクライアントから受け付けず、常に 'contractor' 固定とする（ホワイトリスト方式）
  - DB 関数（complete_registration）を RPC で呼び出し、トランザクション内でユーザー情報・職種・エリアを一括登録する
  - Supabase Auth のパスワードを更新する（一時パスワードからユーザーが入力した本パスワードに上書き）
  - トランザクション成功後に登録完了メールを送信する（Resend 経由。送信失敗時もロールバックしない）
  - トランザクション失敗時はエラーメッセージを返し、ユーザーに再試行を促す（auth.users は削除しない）
  - _Requirements: 6.1_
  - _Contracts: CompleteRegistrationAction Service_

- [x] 7. パスワードリセット機能
- [x] 7.1 (P) パスワードリセット申請画面と Server Action の実装
  - メールアドレス入力フォームを作成する（Client Component）
  - Server Action でサーバー側バリデーション後、Supabase Auth の resetPasswordForEmail を呼び出す
  - リダイレクト先を Auth Callback エンドポイントに設定する
  - 送信成功後に「リセットメールを送信しました」メッセージを表示する
  - 存在しないメールアドレスでも同じメッセージを表示する（アカウント存在推測の防止）
  - _Requirements: 3.1_
  - _Contracts: ResetPasswordAction Service_

- [x] 7.2 (P) パスワード再設定画面と Server Action の実装
  - 新しいパスワードとパスワード確認の2フィールド入力フォームを作成する（Client Component）
  - Server Action でサーバー側バリデーション（8文字以上 + 一致確認）後、Supabase Auth の updateUser でパスワードを更新する
  - 再設定成功時は完了メッセージを表示し、ログイン画面（AUTH-002）へ遷移する
  - トークンが無効または期限切れの場合はエラーメッセージを表示し、パスワードリセット申請画面（AUTH-003）への再申請を案内する
  - _Requirements: 4.1_
  - _Contracts: UpdatePasswordAction Service_

- [x] 8. (P) ランディングページの実装
  - ビジ友のサービス概要を表示するページを作成する（Server Component）
  - 「ログイン」ボタン（AUTH-002 へ遷移）と「新規登録」ボタン（AUTH-005 へ遷移）を配置する
  - フッターによくある質問（COM-007）、お問い合わせ（COM-008）、利用規約（COM-009）、プライバシーポリシー（COM-010）、特定商取引法（COM-011）へのリンクを配置する
  - 認証済みユーザーのリダイレクトは Middleware（タスク2）で処理済み
  - _Requirements: 1.1_

- [x] 9. (P) 登録完了ページの実装
  - 登録完了メッセージを表示するページを作成する（Server Component）
  - 表示後にマイページ（CON-001）へ自動遷移する（再ログイン不要。Supabase Auth のセッションが確立済み）
  - _Requirements: 7.1_

- [x] 10. マイページのメニュー表示分岐の実装
  - Server Component で users.role と subscriptions.status を取得し、表示メニューを決定する
  - 受注者メニュー（11項目: 募集案件一覧、発注者一覧、マイリスト、メッセージ/スカウト一覧、応募履歴一覧、空き日程一覧、本人確認・CCUS登録、プロフィール、有料プラン案内、よくある質問、お問い合わせ）を常時表示する
  - 発注者メニュー（8項目: 募集現場一覧、ユーザー一覧、応募一覧、発注履歴一覧、メッセージ一斉送信、スカウトテンプレート、発注者情報、担当者一覧）を課金済みユーザーにのみ追加表示する
  - 担当者一覧は法人プランのみ表示する（subscriptions.plan_type で判定）
  - 表示判定: role が client または staff、かつ subscriptions.status が active または past_due の場合に発注者メニューを表示する
  - 各メニュー項目のリンク先は画面ID に対応するルートに設定する（遷移先画面は他 spec で実装するため、現時点ではリンク先パスのみ設定）
  - _Requirements: 8.1_

- [x] 11. テスト
- [x] 11.1 ユニットテスト（Zod スキーマ + Middleware ルーティング）
  - 全 Zod バリデーションスキーマ（ログイン、メール認証、パスワードリセット、パスワード再設定、プロフィール入力）の正常系・異常系をテストする
  - Middleware のルーティングルールをロール別にテストする（未認証・受注者・発注者・担当者・管理者の各ロールでアクセス可否を検証）
  - Server Actions のバリデーションとエラーハンドリングをテストする
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 11.2 (P) 統合テスト（RLS ポリシー + complete_registration RPC）
  - users テーブルの RLS テスト: 認証済みユーザーが自分の users レコードのみ更新できることを検証する
  - users テーブルの RLS テスト: INSERT 時に auth.uid() = id の制約が正しく機能することを検証する
  - complete_registration RPC のトランザクションテスト: 正常系（3テーブルが正しく更新される）を検証する
  - complete_registration RPC のトランザクションテスト: 部分失敗時にロールバックが正しく機能することを検証する
  - pgTAP テストとして Supabase テストディレクトリに配置する
  - _Requirements: 6.1_

- [ ] 11.3 (P) E2E テスト（新規登録 + ログインフロー）
  - 新規登録フロー: メール認証 → プロフィール入力 → 登録完了 → マイページ遷移の一連のフローをテストする
  - ログインフロー: メール + パスワード入力 → マイページ表示をテストする
  - Playwright で実装し、E2E テストディレクトリに配置する
  - _Requirements: 2.1, 5.1, 6.1, 7.1, 8.1_
