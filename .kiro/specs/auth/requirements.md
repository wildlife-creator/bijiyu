# 認証機能（auth）— 要件定義

## 概要

ビジ友のユーザー認証基盤。新規登録・ログイン・パスワード管理を提供する。
Supabase Auth を活用し、メール/パスワード認証を実装する。

## 対象画面

| 画面ID | 画面名 | 概要 |
|--------|--------|------|
| AUTH-001 | ログイン前TOP | ランディングページ。ログイン・新規登録・FAQ等への導線 |
| AUTH-002 | ログイン | メールアドレス + パスワード |
| AUTH-003 | パスワードリセット申請 | メールアドレス入力 → リセットメール送信 |
| AUTH-004 | パスワード再設定 | 新しいパスワード入力。完了後: 完了メッセージ → ログイン画面 |
| AUTH-005 | 新規会員登録メール認証 | メールアドレス入力 → 認証メール送信 |
| AUTH-006 | 新規会員登録情報入力 | 基本情報の入力フォーム |
| AUTH-007 | 新規会員情報登録完了 | 登録完了通知。完了後: マイページ（CON-001）へ直接遷移 |

## 対象ロール

- 未認証ユーザー（全画面）
- 認証済みユーザー（ログアウト後の再ログイン）

## 機能要件

### REQ-AUTH-001: ランディングページ（AUTH-001）

- ビジ友のサービス概要を表示する
- 「ログイン」ボタン → AUTH-002 へ遷移
- 「新規登録」ボタン → AUTH-005 へ遷移
- フッターリンク: よくある質問（COM-007）、お問い合わせ（COM-008）、利用規約（COM-009）、プライバシーポリシー（COM-010）、特定商取引法（COM-011）
- 認証済みユーザーがアクセスした場合: CON-001（マイページ）にリダイレクト

### REQ-AUTH-002: ログイン（AUTH-002）

- メールアドレスとパスワードでログインする
- Supabase Auth の `signInWithPassword` を使用
- バリデーション:
  - メールアドレス: 必須、メール形式
  - パスワード: 必須
- ログイン成功時: CON-001（マイページ）へ遷移
- ログイン失敗時: 「メールアドレスまたはパスワードが正しくありません」を表示（具体的なエラー理由は開示しない）
- 「パスワードをお忘れの方」リンク → AUTH-003 へ遷移
- 「新規登録はこちら」リンク → AUTH-005 へ遷移
- 管理者（role = 'admin'）がログインした場合: ADM-002（管理者トップ）へ遷移

### REQ-AUTH-003: パスワードリセット申請（AUTH-003）

- メールアドレスを入力してリセットメールを送信する
- Supabase Auth の `resetPasswordForEmail` を使用
- バリデーション:
  - メールアドレス: 必須、メール形式
- 送信成功時: 「リセットメールを送信しました」を表示
- 存在しないメールアドレスでも同じメッセージを表示（アカウント存在確認を防止）
- リセットメールにはトークン付きURLが含まれる（Supabase Auth 標準）

### REQ-AUTH-004: パスワード再設定（AUTH-004）

- リセットメールのリンクからアクセスする画面
- 新しいパスワードと確認入力の2フィールド
- Supabase Auth の `updateUser` を使用
- バリデーション:
  - 新しいパスワード: 必須、8文字以上
  - パスワード確認: 新しいパスワードと一致
- 再設定成功時: 完了メッセージを表示 → AUTH-002（ログイン画面）へ遷移
- トークンが無効または期限切れの場合: エラーメッセージを表示し、AUTH-003 への再申請を案内

### REQ-AUTH-005: 新規会員登録メール認証（AUTH-005）

- メールアドレスを入力して認証メールを送信する
- Supabase Auth の `signUp` を使用（メール確認を必須に設定）
- バリデーション:
  - メールアドレス: 必須、メール形式、既存アカウントと重複していないこと
- 送信成功時: 「認証メールを送信しました。メールのリンクをクリックしてください」を表示
- 既に登録済みのメールアドレスの場合: セキュリティ上、同じメッセージを表示（存在確認を防止）

### REQ-AUTH-006: 新規会員登録情報入力（AUTH-006）

- メール認証完了後にアクセスする画面
- 入力フィールド:
  - 姓（必須）
  - 名（必須）
  - 性別（必須、選択肢: OptionSets から取得）
  - 生年月日（必須）
  - 都道府県（必須、選択肢: OptionSets から取得）
  - 会社名/屋号（任意）
  - 職種（必須、最大3つ選択、選択肢: OptionSets から取得）
  - 経験年数（職種ごとに設定）
  - 対応可能エリア（必須、複数選択可、都道府県リスト）
  - パスワード設定（必須、8文字以上）
- Server Action 内で以下を実行:
  1. Supabase Auth のセッションからユーザーID を取得（メール認証完了済み = auth.users にレコードが存在する状態）
  2. 以下をトランザクション（一連の処理をまとめて、途中で失敗したら全部取り消す仕組み）で実行:
     a. public.users テーブルを更新（last_name, first_name, gender, birth_date, prefecture, company_name 等を設定）
     b. user_skills テーブルに職種データを INSERT（最大3件）
     c. user_available_areas テーブルに対応可能エリアを INSERT
  3. トランザクション失敗時: エラーメッセージを表示し、ユーザーに再試行を促す（auth.users は削除しない — メール認証は完了しているため、情報入力のリトライのみ）
  4. 登録完了メール送信 — メール送信失敗時は本体処理をロールバックしない（security.md の共通方針に準拠）
- 初期ロール: 'contractor'（受注者）— auth.users 作成時の Database Trigger で public.users が自動生成される際に設定

**role のセキュリティ制限（三重防御）:**
- **Server Action:** role パラメータはクライアントから受け付けない。常に `'contractor'` を固定値として設定する（ホワイトリスト方式）。万が一リクエストに role が含まれていても無視する
- **RLS:** users テーブルの INSERT ポリシーで `role = 'contractor'` のみ許可する。`role = 'admin'` での INSERT はRLSによりブロックされる
- **理由:** admin ロールはシステム管理者専用であり、SQL Editor からのみ作成可能（specs/admin/requirements.md 参照）。一般ユーザーの登録フローで admin が作成されることは絶対に防ぐ必要がある

### REQ-AUTH-007: 新規会員情報登録完了（AUTH-007）

- 登録完了メッセージを表示する
- 登録完了後: CON-001（マイページ）へ直接遷移（再ログイン不要）
- Supabase Auth のセッションが自動的に確立された状態

### REQ-AUTH-008: マイページ（CON-001）

- 認証済みユーザーのトップページ（ダッシュボード）
- ログイン後・新規登録後の遷移先
- 受注者メニュー（常に表示）:
  - 募集案件一覧（CON-002）
  - 発注者一覧（CON-005）
  - マイリスト（CON-007）
  - メッセージ/スカウト一覧（CON-008）
  - 応募履歴一覧（CON-011）
  - 空き日程一覧（CON-014）
  - 本人確認・CCUS登録（COM-003）
  - プロフィール（COM-001）
  - 有料プラン案内（CLI-026）
  - よくある質問（COM-007）、お問い合わせ（COM-008）等
- 発注者メニュー（課金後に追加表示）:
  - 募集現場一覧（CLI-001）
  - ユーザー一覧/職人一覧（CLI-005）
  - 応募一覧（CLI-007、未対応 status='applied' のみ）
  - 発注履歴一覧（CLI-010、発注可否決定以降 status≠'applied'）
  - メッセージ一斉送信（CLI-014）
  - スカウトテンプレート（CLI-016）
  - 発注者情報（CLI-020）
  - 担当者一覧（CLI-022、法人プランのみ）
- 表示判定: users.role と subscriptions.status で分岐
- CON-001 は spec 横断の共通エントリポイント

## 非機能要件

### セキュリティ

- パスワードは Supabase Auth が安全にハッシュ化して保管（アプリ側では生パスワードを保持しない）
- ログイン失敗時にアカウント存在の有無を推測させない
- セッション管理は Supabase Auth のJWTトークン + httpOnly Cookie
- パスワードリセットトークンは時間制限付き（Supabase Auth 標準）
- 全フォーム入力に Zod バリデーション（クライアント + サーバー）

### 通知メール

- メール送信失敗で本体処理をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）

### パフォーマンス

- ログイン処理は2秒以内に完了すること
- メール送信は非同期処理（ユーザーの待ち時間に影響しない）

## 画面遷移

```
AUTH-001 → AUTH-002（ログイン）→ CON-001（マイページ）
AUTH-001 → AUTH-005（メール認証）→ AUTH-006（情報入力）→ AUTH-007（完了）→ CON-001
AUTH-002 → AUTH-003（リセット申請）→ AUTH-004（再設定）→ AUTH-002
```
