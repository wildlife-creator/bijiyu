# メッセージ機能（messaging）— 要件定義

## 概要

受注者と発注者間の1対1メッセージ、スカウト送信、一斉送信を提供する機能。
Supabase Realtime を使用したリアルタイム通信を実装する。

**スレッドモデル**: 1組織（or 個人発注者）× 1受注者 = 常に1スレッド。法人プランの場合、同一組織のメンバー全員がスレッドを共有する。スカウトメッセージも通常メッセージと同じスレッド内に入る。

## 対象画面

| 画面ID | 画面名 | ロール | 概要 |
|--------|--------|--------|------|
| CON-008 | メッセージ/スカウト一覧 | 受注者・発注者 | スレッド一覧 |
| CON-009 | メッセージ/スカウト詳細 | 受注者・発注者 | 個別のやり取り画面 |
| CON-010 | メッセージ入力/送信 | 受注者・発注者 | メッセージ入力・送信 |
| CLI-013 | メッセージ詳細 | 発注者 | ユーザー詳細からの直接メッセージ |
| CLI-014 | メッセージ一斉送信 | 発注者 | 複数の職人へのメッセージ一斉送信 |
| CLI-015 | スカウト送信 | 発注者 | 職人へのスカウトメッセージ送信 |

## 対象ロール

- 受注者（Contractor）: メッセージ受信・返信は無制限。新規メッセージは月5通制限
- 発注者（Client）: メッセージ無制限、一斉送信・スカウト送信可能
- 担当者（Staff）: 組織内のメッセージを操作可能。**同一組織のスレッドを全て閲覧・送信可能**

## 機能要件

### REQ-MSG-001: メッセージ/スカウト一覧（CON-008）

- 自分が参加しているスレッド、および自分の組織のスレッドの一覧を表示する
- 法人プラン: 組織メンバー全員のスレッドが表示される（組織としてのやり取り一覧）
- 個人プラン: 自分が participant のスレッドのみ表示される（従来型）
- 表示項目: 相手の名前、アイコン、最新メッセージのプレビュー、日時、未読バッジ
- スレッド種別: message（通常メッセージのみ）/ scout（スカウトメッセージを含む）
- タブ切り替えまたはフィルターでメッセージとスカウトを分けて表示
- ソート: 最新メッセージ日時の降順
- 未読バッジ: Supabase Realtime で即時更新
- 各スレッドクリック → CON-009 へ遷移
- **一斉送信ボタン**: 発注者（client）・担当者（staff）のみ表示。受注者（contractor）の場合は非表示。ロールによる条件分岐で同一コンポーネントを共用する

### REQ-MSG-002: メッセージ/スカウト詳細（CON-009）

- 個別スレッドのメッセージ一覧を時系列で表示する
- 1スレッド内に通常メッセージとスカウトメッセージが混在する
- 表示項目: 送信者アイコン、メッセージ本文、添付画像、送信日時、既読マーク
- **スカウトメッセージの表示**（is_scout = true のメッセージ）:
  - メッセージバブル内にスカウト案件情報カード（job_id に紐づく案件タイトル、報酬、エリア、募集期間等）をインライン表示
  - 各スカウトメッセージごとに `scout_status` に応じた受諾/拒否ボタンを表示
- 代理メッセージ: is_proxy = true のメッセージに「代理」バッジを表示（**発注者側の画面のみ**。受注者側には表示しない）
- Supabase Realtime:
  - 新着メッセージをリアルタイムで受信・表示（INSERT イベントのみ購読）
  - 購読チャネル: `messages:thread_id={スレッドID}` でフィルタリング
  - 画面アンマウント時に購読解除（メモリリーク防止）
  - 接続断→自動再接続後、差分メッセージを取得（最後の受信タイムスタンプ以降）
- 既読管理: メッセージ表示時に read_at を更新（デバウンス処理で頻度制限、**推奨: 3〜5秒**。ユーザーがスクロールするたびにサーバーへ問い合わせるのではなく、スクロールが止まってから3〜5秒後にまとめて1回だけ既読更新リクエストを送る）
- メッセージ入力エリア → CON-010 と統合（同一画面内で入力・送信）
- **スカウト受諾/拒否フロー**（is_scout = true のメッセージごと）:
  - `messages.scout_status` に応じた表示制御:
    - `scout_status = 'pending'`: 「スカウトを受ける」「スカウトを断る」ボタンをスカウトメッセージに隣接して表示（受注者側 CON-009 のみ）
    - `scout_status = 'accepted'`: ボタンの代わりに「スカウトを受けました」テキストを表示（受注者・発注者の両方に表示）
    - `scout_status = 'rejected'`: ボタンの代わりに「スカウトを断りました」テキストを表示（受注者・発注者の両方に表示）
  - 「スカウトを受ける」ボタン押下時:
    1. `messages.scout_status` を `'accepted'` に更新（Server Action）
    2. CON-004（応募情報入力）へ遷移: `/applications/new?job_id={jobId}&scout_message_id={messageId}`
    3. 応募データ保存時に `applications.scout_message_id` にスカウトメッセージIDが記録される（matching 側で実装）
  - 「スカウトを断る」ボタン押下時:
    1. 確認ダイアログを表示（「スカウトを断りますか？」）
    2. `messages.scout_status` を `'rejected'` に更新（Server Action）
    3. 「スカウトを断りました」テキストに切り替わる（状態反映）
  - 拒否後のスレッド状態: メッセージの送受信は引き続き可能。スレッドは通常のメッセージスレッドとして機能し続ける
- **法人プラン**: 同一組織の担当者全員がスレッドを閲覧・メッセージ送信可能

### REQ-MSG-003: メッセージ入力/送信（CON-010）

- メッセージを入力・送信する
- 入力フィールド:
  - メッセージ本文（必須）
  - 画像添付（任意、JPEG/PNG、最大10MB）
- 楽観的UI:
  - 送信ボタン押下 → 即座にUIに反映 → Server Action 実行
  - 失敗時: UIをロールバック + エラートースト表示
- messages テーブルにデータを保存
- **代理アカウントからの送信（法人プランのみ）**:
  - 代理アカウントとは: **ビジ友の運営スタッフ**（サービス提供側の人間）が、法人の発注者に代わってメッセージ対応を行うためのアカウント。法人組織内に担当者として作成し（`is_proxy_account = true`）、運営スタッフがそのアカウントにログインして操作する
  - 代理アカウントは通常の担当者と同じフローでメッセージを送信する。sender_id の書き換えや特別な権限チェックは不要
  - メッセージ送信時、Server Action が送信者の `organization_members.is_proxy_account` を参照し、`true` の場合は `messages.is_proxy = true` を自動設定する
  - 発注者側の画面では `is_proxy = true` のメッセージに「代理」バッジを表示し、代理アカウントからの送信であることを識別可能にする。受注者側には「代理」バッジを表示しない（受注者には企業名のみ表示される）
  - 代理メッセージの通数上限（年36通/300通）はシステムでは制御しない。プラン説明上の目安として記載するのみ（運用管理。roles-and-permissions.md 参照）
  - 代理アカウントの設定権限: is_proxy_account フラグを設定できるのは管理責任者（owner）と管理者（admin）のみ（設定は CLI-024/025 の担当者編集・新規作成画面で行う。REQ-ORG-009/010 参照）
- **無料ユーザーの送信制限**:
  - 新規メッセージ（まだやり取りしたことがない相手への初回メッセージ）: 月5通まで
  - 返信（既存スレッドへのメッセージ）: 制限なし
  - 「当月」の定義: **カレンダー月（JST基準）**（1月1日〜1月31日、2月1日〜2月28日…）。毎月1日の JST 0:00 にカウントがリセットされる
  - カウントはサーバー側で実施（当月の新規スレッド作成数をカウント）。Supabase の timestamptz は UTC 保存のため、カウント時に JST に変換する。参考SQL: `WHERE created_at >= date_trunc('month', NOW() AT TIME ZONE 'Asia/Tokyo') AT TIME ZONE 'Asia/Tokyo'`
  - 制限に達した場合: エラーメッセージ「今月の新規メッセージ上限に達しました。有料プランにアップグレードすると無制限で送信できます」
- レート制限: 同一ユーザーから1分間に3通まで。Server Action 内で `messages` テーブルの `(sender_id, created_at)` インデックスを使って直近1分間の送信数をカウントし、超過時はエラーを返す

### REQ-MSG-004: メッセージ詳細 — 発注者（CLI-013）

- ユーザー詳細（CLI-006）から直接メッセージを送信する画面
- 既存のスレッドがある場合: そのスレッドのメッセージ詳細を表示
  - 法人プラン: 同一組織の誰かが既にスレッドを持っている場合、そのスレッドを表示（organization_id で検索）
  - 個人プラン: participant_1_id + participant_2_id で検索
- 既存のスレッドがない場合: 新規スレッドを作成してメッセージを送信
  - 法人プラン: organization_id を設定してスレッド作成
  - 個人プラン: participant_1_id = 自分、participant_2_id = 相手
- **CON-009 / CON-010 との関係:** CLI-013 は CON-009（メッセージ詳細表示）+ CON-010（メッセージ入力・送信）と同じコンポーネント（部品）を再利用して作る別画面。入り口（遷移元）が異なるだけで、中身の機能は同じ
  - CON-009 → メッセージ一覧（CON-008）から遷移
  - CLI-013 → ユーザー詳細（CLI-006）から遷移
- **CLI-013 ではスカウト受諾/拒否ボタンを表示しない**: CON-009 と同じコンポーネントを再利用するが、「スカウトを受ける」「スカウトを断る」ボタンは CLI-013（発注者側）では非表示とする。スカウトへの応答は受信者（受注者側 = CON-009）のみが行える。ただし、スカウト応答後のステータステキスト（「スカウトを受けました」「スカウトを断りました」）は CLI-013（発注者側）にも表示する

### REQ-MSG-005: メッセージ一斉送信（CLI-014）

- 複数の職人に同一メッセージを一括送信する（発注者専用）
- 入力フィールド:
  - 送信先の選択（職人一覧からチェックボックスで選択）
  - 「全選択」/「全解除」ボタン: 送信先を一括で選択・解除できる
  - メッセージ本文（必須）
- 送信先ごとにスレッドを検索（既存スレッドに追加 or 新規作成）し、メッセージを保存
  - 法人プラン: organization_id で既存スレッドを検索
  - 個人プラン: participant_1_id + participant_2_id で検索
- 一斉送信の進捗表示
- 送信完了後: 完了メッセージを表示

### REQ-MSG-006: スカウト送信（CLI-015）

- 職人にスカウトメッセージを送信する（発注者専用）
- ユーザー詳細（CLI-006）から遷移
- 入力フィールド:
  - 案件の選択（自社の掲載中案件から選択、job_id を設定）
  - メッセージ本文（必須）
  - テンプレートから選択可能（scout_templates から読み込み）
- **スレッドの作成/再利用**:
  - 対象受注者との既存スレッドがある場合: そのスレッドにスカウトメッセージを追加
  - 既存スレッドがない場合: 新規スレッドを作成（法人プランでは organization_id を設定）
  - スレッドの thread_type を 'scout' に更新（まだ 'message' の場合）
- messages テーブルに is_scout = true、scout_status = 'pending'、job_id 設定で保存
- 同一受注者 × 同一案件で既にスカウトメッセージがある場合はエラー（重複防止）
- 受注者へメール通知（スカウト受信通知）

## 非機能要件

### セキュリティ

- RLS: メッセージは送信者・受信者のみ閲覧可能。法人プランの場合は同一組織メンバーも閲覧可能
- 一斉送信: 発注者プランのみ許可（RLSで制御）
- スカウト送信: 発注者プランのみ許可
- メッセージ内URLのサニタイズ（XSS防止）
- 外部URL表示時は警告を表示
- 画像は Supabase Storage にUUIDファイル名で保存

### パフォーマンス

- Supabase Realtime の同時接続数: Freeプラン 200接続を考慮
- 購読チャネルはフィルタリングして不要データの受信を防止
- 既読更新はデバウンス処理で頻度を制限

### 通知

- 新着メッセージ通知: 未読一定時間後にメール送信（Resend）
- スカウト受信通知: 即時メール送信
- 通知バッジ（未読数）: Supabase Realtime で即時更新
- メール送信失敗時: 本体処理（メッセージ保存・スレッド作成）をロールバックしない。詳細は security.md の「メール送信失敗時の共通方針」を参照

### 名前表示ルール（メッセージ UI・メール通知共通）

メッセージ画面およびメール通知で「相手の名前」を表示する際の優先順位を以下に定義する。このルールはメッセージ機能だけでなく、マッチング通知メール等の他機能のメールテンプレートでも共通で使用する。

#### 発注者側の名前を表示する場合（受注者が見る相手の名前）

法人プランの発注者（organization あり）:
```
organizations.name（企業名）
```
→ 担当者（staff）が送信した場合も、受注者には組織名が表示される。個人の `users.company_name` は使わない（複数人が利用する法人プランでは、組織として統一された名前を表示するため）。

個人・小規模プランの発注者（organization なし）:
```
1. users.company_name（屋号 — COM-002 で入力）
2. users.last_name + first_name（フォールバック）
```

#### 受注者側の名前を表示する場合（発注者が見る相手の名前）

```
1. users.company_name（屋号 — COM-002 で入力）
2. users.last_name + first_name（フォールバック）
```
受注者は組織を持たないため `organizations.name` は常に null。

#### メール宛名（greeting）

全ユーザー共通で個人名を使用:
```
users.last_name + first_name
```

#### 名前解決ユーティリティ

共通の名前解決関数を用意し、メッセージ UI・メール通知・マッチング通知で統一的に使用する。既存の `src/lib/utils/display-name.ts` を拡張するか、新規に `resolveDisplayName()` 関数を作成する。

#### 現在の実装との差分

- **メッセージ UI（スレッド一覧・詳細）**: 法人は `organizations.name` を使用（実装済み）。個人・小規模の場合 `users.company_name` へのフォールバックが未実装 → 追加が必要
- **メッセージ通知メール**: `senderName` パラメータに上記の名前解決結果を渡す → Server Action 修正が必要
- **スカウト通知メール**: 同上
- **マッチング通知メール**: `clientName` がハードコード `"発注者"` → 名前解決結果に変更（matching spec で対応）
- **応募履歴画面**: `organizations.name ?? owner.company_name` のフォールバックは実装済み（参考パターン）

### 設計上の制約（Non-Goals）

- **メッセージの削除・編集は提供しない**: 建設業マッチングサービスにおける業務連絡の証跡保全のため、送信済みメッセージの取り消し・編集機能は意図的に対象外とする
- **スカウトテンプレートの CRUD（CLI-016〜019）は messaging スコープ外**: organization spec で実装する。CLI-015（スカウト送信）からの「テンプレートから選択」は scout_templates テーブルの SELECT のみ行う
