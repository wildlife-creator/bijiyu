# マッチング機能（matching）— 要件定義

## 概要

応募から発注可否判断、作業完了/失注報告、相互評価までの一連のフローを管理する機能。
受注者側の応募履歴管理と、発注者側の応募管理・発注履歴管理を含む。

## 対象画面

| 画面ID | 画面名 | ロール | 概要 |
|--------|--------|--------|------|
| CON-011 | 応募履歴一覧 | 受注者 | 自分の応募案件のリスト |
| CON-012 | 応募詳細 | 受注者 | 応募した案件の詳細・ステータス表示 |
| CON-013 | 作業完了/失注報告・評価登録 | 受注者 | 受注者側の完了報告 + 発注者への評価 |
| CLI-007 | 応募一覧 | 発注者 | 案件への応募者リスト |
| CLI-008 | 応募詳細 | 発注者 | 応募者のプロフィール・発注ボタン |
| CLI-009 | 発注可否 | 発注者 | 発注の承認/拒否 |
| CLI-010 | 発注履歴一覧 | 発注者 | 発注済み案件のリスト |
| CLI-011 | 発注履歴詳細 | 発注者 | 発注済み案件の詳細・ステータス |
| CLI-012 | 作業完了/失注報告・評価登録 | 発注者 | 発注者側の完了報告 + 受注者への評価（6項目） |
| CLI-028 | 発注者評価 | — | 受注者→発注者の評価表示画面 |

## 対象ロール

- 受注者（Contractor）: CON-011〜013
- 発注者（Client）: CLI-007〜012, CLI-028
- 担当者（Staff）: 組織内の案件について CLI-007〜012 を操作可能

## 機能要件

### 受注者側

#### REQ-MT-001: 応募履歴一覧（CON-011）

- 自分が応募した案件の一覧を表示する
- 表示項目: 案件タイトル、発注者名、ステータス、応募日
- ステータス: 応募中 / 発注済み / お断り / 完了 / キャンセル / 失注
- ソート: 応募日の降順
- ページネーション: 20件ずつ
- 各応募クリック → CON-012 へ遷移

#### REQ-MT-002: 応募詳細（CON-012）

- 応募した案件の詳細情報を表示する
- 表示項目:
  - 案件情報（タイトル、職種、報酬、勤務地、工期）
  - 自分の応募情報（応募人数、日程、初回稼働希望日、申し送り）
  - ステータスバッジ
  - 初回稼働日（確定後に表示）
- ステータスが 'applied'（応募中）の場合:
  - 「応募をキャンセルする」ボタンを表示
  - ただし、初回稼働希望日（preferred_first_work_date）の5日前を過ぎた場合はボタンを非活性にし、注意文言を表示:
    「初回稼働日の5日前を過ぎたため、システムからはキャンセルできません。キャンセルが必要な場合は、発注者にメッセージでご連絡ください。」
  - キャンセルボタン押下時: 確認ポップアップ → applications.status を 'cancelled' に更新
  - キャンセル完了後: CON-011（応募履歴一覧）へ遷移
- ステータスが 'accepted'（発注済み）の場合:
  - 「作業完了/失注報告」ボタン → CON-013 へ遷移
  - キャンセルボタンは非表示（発注済みの応募は受注者からキャンセル不可。発注者に連絡して対応してもらう）

#### REQ-MT-003: 作業完了/失注報告・評価登録 — 受注者側（CON-013）

- 受注者が作業完了または失注を報告し、発注者を評価する
- 入力フィールド:
  - 稼働状況（必須、'completed' / 'lost'）
  - 稼働状況の補足（任意）
  - また仕事を受けたいか（必須、'good' / 'bad'）— client_reviews の唯一の評価項目
  - 評価の補足コメント（任意）
- applications テーブルの status を更新（'completed' or 'lost'）
- client_reviews テーブルにデータを保存
- 保存成功後: CON-011（応募履歴一覧）へ遷移

### 発注者側

#### REQ-MT-004: 応募一覧（CLI-007）

- 自社の案件に対する応募者の一覧を表示する
- 案件ごとにフィルター可能
- 表示項目: 応募者名、職種、応募日、ステータス
- ページネーション: 20件ずつ
- 各応募クリック → CLI-008 へ遷移

#### REQ-MT-005: 応募詳細（CLI-008）

- 応募者のプロフィール情報と応募内容を表示する
- 表示項目:
  - 応募者プロフィール（アイコン、氏名、職種、経験年数、スキル、本人確認バッジ、CCUSバッジ）
  - 応募情報（応募人数、日程、初回稼働希望日、申し送り）
  - 応募者の評価履歴（user_reviews からの集計）
- 「発注する/お断りする」ボタン → CLI-009 へ遷移

#### REQ-MT-006: 発注可否（CLI-009）

- 応募に対して発注（承認）またはお断り（拒否）を行う
- 「発注する」ボタン:
  - applications.status を 'accepted' に更新
  - applications.first_work_date を設定（初回稼働日）
  - 受注者へメール通知（マッチング成立通知）
  - 受注者の応募履歴ステータスも更新
- 「お断りする」ボタン:
  - applications.status を 'rejected' に更新
  - 受注者へメール通知（お断り通知）
- 完了後: ポップアップ「ユーザーへ結果を送信しました」→ CLI-007（応募一覧）へ遷移

#### REQ-MT-007: 発注履歴一覧（CLI-010）

- 発注済み（status = 'accepted' / 'completed' / 'lost'）の案件一覧を表示する
- 表示項目: 案件タイトル、受注者名、ステータス、発注日
- ページネーション: 20件ずつ
- 各履歴クリック → CLI-011 へ遷移

#### REQ-MT-008: 発注履歴詳細（CLI-011）

- 発注済み案件の詳細情報を表示する
- 表示項目:
  - 案件情報、受注者情報、初回稼働日
  - ステータスバッジ
  - 評価情報（登録済みの場合）
- ステータスが 'accepted'（発注済み）の場合:
  - 「作業完了/失注報告」ボタン → CLI-012 へ遷移

#### REQ-MT-009: 作業完了/失注報告・評価登録 — 発注者側（CLI-012）

- 発注者が作業完了または失注を報告し、受注者を評価する
- 入力フィールド:
  - 稼働状況（必須、'completed' / 'lost'）
  - 稼働状況の補足（任意）
  - 評価6項目（各 必須、'good' / 'bad'）:
    1. また依頼したいか（rating_again）
    2. 指示通りに動けるか（rating_follows_instructions）
    3. 稼働予定日にちゃんと来たか（rating_punctual）
    4. 作業は速いか（rating_speed）
    5. 作業は丁寧か（rating_quality）
    6. 工事に必要な工具を持っているか（rating_has_tools）
  - 評価の補足コメント（任意）
- applications テーブルの status を更新
- user_reviews テーブルにデータを保存
- 保存成功後: CLI-010（発注履歴一覧）へ遷移

### 評価表示

#### REQ-MT-010: 発注者評価表示（CLI-028）

- ユーザー詳細（CLI-006）から遷移して、受注者→発注者の評価を表示する
- 表示項目:
  - 「また仕事を受けたいか」の Good / Bad 集計
  - 評価の補足コメント一覧
- 現時点では1項目のみ（将来的に評価軸追加の可能性あり）

## 非機能要件

### セキュリティ

- Middleware: 受注者は CON 系のみ、発注者は CLI 系のみアクセス可能
- RLS:
  - 受注者: 自分の応募のみ閲覧可能
  - 発注者: 自分（または所属組織）の案件への応募のみ閲覧可能
  - 評価: 作成は当事者のみ、閲覧は公開
- メール通知: 発注/お断り結果の通知はサーバーサイドから送信
- メール送信失敗で本体処理をロールバックしない（security.md の「メール送信失敗時の共通方針」に準拠）

### データ整合性

- 評価は1応募につき1件のみ（UNIQUE (application_id) — 評価者は応募ごとに1人に確定するため application_id のみで一意性を保証）
- **評価の更新・削除不可**: 一度投稿した評価（user_reviews, client_reviews）は編集・削除できない。これは評価の信頼性を保つための意図的な制約であり、UI上に編集・削除ボタンは配置しない
- ステータス遷移の整合性:
  - applied → accepted（発注者が発注）
  - applied → rejected（発注者がお断り）
  - applied → cancelled（受注者が自分でキャンセル、またはユーザー退会時の自動キャンセル）
  - accepted → completed（作業完了報告）
  - accepted → lost（失注報告）
  - accepted → cancelled（管理者による発注取り消し — ADM-014。受注者自身はキャンセル不可）
  - **受注者キャンセルの制約**: status = 'applied' かつ preferred_first_work_date - 5日 > NOW() の場合のみ可能
  - **再応募の制約**: rejected 後の同一案件への再応募は不可。DB の UNIQUE 制約 `(job_id, applicant_id) WHERE status NOT IN ('cancelled')` により、rejected レコードが存在する場合は DB レベルで重複 INSERT がブロックされる（cancelled のみ除外 = rejected は制約に含まれるため再応募不可）。cancelled 後の再応募は可能（UNIQUE 制約から除外されているため）

## 画面遷移

```
受注者:
CON-001 → CON-011（応募履歴一覧）→ CON-012（応募詳細）→ CON-013（完了報告・評価）

発注者:
CON-001 → CLI-007（応募一覧）→ CLI-008（応募詳細）→ CLI-009（発注可否）→ CLI-007
CON-001 → CLI-010（発注履歴一覧）→ CLI-011（発注履歴詳細）→ CLI-012（完了報告・評価）
CLI-006（ユーザー詳細）→ CLI-028（発注者評価表示）
```

## 関連テーブル

- applications: 応募情報（CRUD）
- user_reviews: 発注者→受注者の評価（6項目）
- client_reviews: 受注者→発注者の評価（1項目）
- jobs: 案件情報（SELECT）
- users: ユーザー情報（SELECT）
