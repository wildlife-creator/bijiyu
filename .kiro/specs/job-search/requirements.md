# 案件検索機能（job-search）— 要件定義

## 概要

受注者（職人）が案件を検索・閲覧し、応募する機能。発注者の一覧検索・詳細閲覧、
発注者側の職人検索、マイリスト（お気に入り）管理も含む。
無料ユーザーは全案件を閲覧可能だが、応募は登録職種×登録県に合致する案件のみ。

## 対象画面

| 画面ID | 画面名 | 概要 |
|--------|--------|------|
| CON-002 | 募集案件一覧 | 案件のリスト表示 |
| CON-003 | 募集案件詳細 | 案件の詳細情報、応募ボタン |
| CON-004 | 応募情報入力 | 応募内容の入力 |
| CON-005 | 発注者一覧 | 発注者の検索結果リスト |
| CON-006 | 発注者詳細 | 発注者の会社情報・評判等の詳細表示 |
| CON-007 | マイリスト | お気に入り案件/発注者/見込みユーザーの管理 |
| CLI-005 | ユーザー一覧（職人一覧） | 発注者が職人を検索・リスト表示 |
| CLI-006 | ユーザー詳細（職人詳細） | 職人のプロフィール詳細 |

## 対象ロール

- 受注者（Contractor / 無料）: 全案件閲覧可。応募は登録職種×登録県のみ
- 発注者（Client / 有料）: 全案件閲覧可。応募に制限なし

> **⚠️ 実装上の注意（全CON系画面共通）**
> CON系画面は受注者専用ではない。発注者・担当者も同じ画面にアクセスし、同じUIを使用する。
> 画面IDの「CON」プレフィックスは「Contractor専用」ではなく「受注者系フロー起点」の意味である。
> ページコンポーネントで role による表示分岐を行ってはならない（応募制限はCON-003の応募ボタンでのみ制御）。

## 機能要件

### REQ-JS-001: 募集案件一覧（CON-002）

- 募集中（status = 'open'）の案件を一覧表示する
- 表示項目: タイトル、職種、勤務地（都道府県）、報酬範囲、急募フラグ、募集期間
- 検索・フィルター:
  - キーワード検索（タイトル、詳細）
  - 職種フィルター（OptionSets から選択）
  - エリアフィルター（都道府県）
  - 報酬範囲フィルター
- ソート: 新着順（デフォルト）、報酬高い順、報酬低い順
- ページネーション: 20件ずつ
- **上位表示ロジック**（デフォルトの新着順ソートに適用）:
  - 第1優先: 急募案件（jobs.is_urgent = true）を最上位に表示
  - 第2優先: 新着順（created_at DESC）
  - 急募案件同士の中でも新着順でソートする
  - SQL例: `ORDER BY is_urgent DESC, created_at DESC`
  - ユーザーが報酬順ソートを選択した場合は上位表示ロジックを適用しない（報酬額のみでソート）
  - 急募案件には一覧上で急募バッジを表示（REQ-JS-002 の急募バッジと同じデザイン）
- お気に入りボタン（♡）: タップで favorites テーブルに登録/解除
- 各案件クリック → CON-003 へ遷移
- **ロール別動作ルール（重要 — リグレッション防止）**:
  - CON-002 は受注者（contractor）・発注者（client）・担当者（staff）の全ロールで **同一のUI・同一のレイアウト** を表示する
  - ページコンポーネントやServer Componentで `role === 'contractor'` による表示分岐をしてはならない
  - 発注者でログインしても、カード表示・ボタン配置・検索フィルターは受注者と完全に同じ
  - 応募制限の違いは CON-003（詳細画面）の応募ボタン活性/非活性でのみ反映される（CON-002 では反映しない）
  - データ取得クエリで `users.role` によるフィルタリングをしてはならない（全ロールで同じ案件一覧を返す）
  - Middleware で CON-002 へのアクセスを contractor のみに制限してはならない

### REQ-JS-002: 募集案件詳細（CON-003）

- 案件の全情報を表示する
- **DB に値が登録されていない項目も空欄（—）で常に表示する**（項目自体を非表示にしない）
- 表示項目（セクション構成）:
  - タイトル + 会社名 + 案件詳細（description）
  - 「条件」セクション: 報酬、エリア、募集職種、募集人数、現場工期、募集期間、稼働時間、締め切り、経験年数、必須スキル、国籍・言語、持ち物
  - 「業務内容」セクション: スケジュール詳細、請負案件詳細
  - 「発注者からのメッセージ」セクション: 発注者メッセージ（枠線付きカード表示）
  - 急募バッジ（該当する場合）
  - 添付画像・書類
  - 発注者情報（会社名、リンク → CON-006）
- お気に入りボタン（♡）
- 「応募する」ボタン → CON-004 へ遷移
- **案件オーナー・同一組織メンバーの応募ボタン非表示**:
  - 案件オーナー（`job.owner_id === currentUser.id`）がCON-003を表示した場合、応募ボタンは非表示とする
  - 同一組織のメンバー（`organization_members` に同じ `organization_id` で所属）がCON-003を表示した場合も、応募ボタンは非表示とする
  - 応募ボタンの非表示判定は、全ての応募制限チェックより前に行う
  - 案件情報の表示内容はオーナー・同一組織メンバーでも変更しない（応募ボタンのみ非表示）
- **応募制限（無料ユーザー）**:
  - 案件の trade_type がユーザーの user_skills.trade_type に含まれ、かつ案件の prefecture がユーザーの user_available_areas.prefecture に含まれる場合のみ応募ボタンが活性
  - 合致しない場合: 応募ボタン非活性 + メッセージ「有料プランに加入するか、プロフィールの職種・エリアを更新してください」
  - 有料ユーザー: 制限なし（常に応募可能）
- **CON-003 と CLI-002 の URL 設計（重要）**:
  - CON-003（募集案件詳細）と CLI-002（募集現場詳細）は同じ案件IDに対して異なるUIを提供する
  - 同一 URL パス `/jobs/[id]` を共有する場合、`?manage=true` クエリパラメータで区別する:
    - パラメータなし（デフォルト）→ CON-003 を表示（閲覧・応募用UI）
    - `?manage=true` → CLI-002 を表示（管理・編集用UI、案件のオーナーまたは同一組織メンバーかつ発注者のみ）
  - CON-002（募集案件一覧）のカードリンクには `?manage=true` を付けない（常に CON-003 へ遷移）
  - CLI-001（募集現場一覧）のカードリンクには `?manage=true` を付ける（常に CLI-002 へ遷移）
  - isOwner だけで表示を分岐してはならない（遷移元に関わらず管理画面になってしまうため）。法人プランでは同一組織メンバーも管理画面を表示できる必要がある
  - 発注者が CON-002 からアクセスした場合は、自分が掲載した案件であっても CON-003（閲覧・応募用UI）を表示する

### REQ-JS-003: 応募情報入力（CON-004）

- 案件への応募情報を入力する
- 入力フィールド:
  - 応募人数（必須、数値）
  - 日程/働き方（必須）
  - 初回稼働希望日（必須、日付）
  - 申し送り（任意、テキスト）
- 入力完了後: ポップアップ確認「この情報で応募して良いですか」
  - OK → applications テーブルにデータ保存（status = 'applied'）
  - OK → 完了ポップアップ「応募が完了しました」を表示 → OK → CON-011（応募履歴一覧）へ遷移
  - 独立した確認/完了画面は存在しない（screen-navigation.md のポップアップ一覧と整合）
- **応募制限のサーバーサイド検証**:
  - Server Action で無料ユーザーの場合、職種×エリアの合致を再検証
  - フロントエンドのバイパス対策
- 同一案件への重複応募を防止（applicant_id + job_id のユニーク制約）

### REQ-JS-004: 発注者一覧（CON-005）

- 発注者（client_profiles）の一覧を表示する
- 表示項目: 会社名/表示名、募集職種、募集エリア、プロフィール画像
- 検索・フィルター:
  - キーワード検索（会社名）— display_name に対する部分一致（ILIKE）
  - 職種フィルター — client_profiles.recruit_job_types 配列に対する重複チェック（`&&` 演算子）。選択した職種のいずれかが recruit_job_types に含まれていればヒット
  - エリアフィルター — client_profiles.recruit_area 配列に対する重複チェック（`&&` 演算子）。選択したエリアのいずれかが recruit_area に含まれていればヒット
- **検索ロジックの詳細（エリア・職種フィルター）**:
  - エリアフィルターと職種フィルターはOR条件で動作する
  - 例: recruit_area = ['東京都', '神奈川県'] の発注者は、エリアフィルターで「神奈川県」を選択した場合にヒットする
  - 例: recruit_job_types = ['大工', '電気工事士'] の発注者は、職種フィルターで「大工」を選択した場合にヒットする
  - SQL例: `WHERE recruit_area && ARRAY['神奈川県']::text[]`（配列の重複チェック）
  - 完全一致（`=`）や文字列の部分一致（`LIKE`）を使ってはならない
- ページネーション: 20件ずつ
- お気に入りボタン（♡）
- 各発注者クリック → CON-006 へ遷移

### REQ-JS-005: 発注者詳細（CON-006）

- 発注者の詳細情報を表示する
- 表示項目:
  - プロフィール画像、会社名/表示名
  - 募集職種、募集エリア、従業員規模
  - 求める働き方、言語、発注者メッセージ（紹介文）
  - 掲載中の案件一覧
- お気に入りボタン（♡）
- 「メッセージを送る」導線（messaging spec へ）

### REQ-JS-006: マイリスト（CON-007）

- お気に入り登録した案件・発注者・ユーザーの一覧を表示する
- プルダウンで表示内容を切り替え:
  - 受注者: お気に入り案件一覧 / お気に入り発注者一覧（2項目）
  - 発注者: 上記 + 見込みユーザー一覧（3項目）
- 各項目クリック時の遷移先:
  - お気に入り案件 → CON-003（募集案件詳細）
  - お気に入り発注者 → CON-006（発注者詳細）
  - 見込みユーザー → CLI-006（ユーザー詳細）
- 「興味あり」（♡ボタン）で登録した内容と同一
- favorites テーブルから target_type でフィルタリング
- お気に入り解除ボタン

### REQ-JS-007: ユーザー一覧 / 職人一覧（CLI-005）— 発注者専用

- 発注者が職人を検索・一覧表示する
- 表示項目: アイコン、氏名、年齢、職種、エリア、経験年数、本人確認バッジ、CCUSバッジ
- 検索・フィルター:
  - キーワード検索（氏名）
  - 職種フィルター（OptionSets）
  - エリアフィルター（都道府県）
- ソート: 新着順（デフォルト）
- ページネーション: 20件ずつ
- お気に入りボタン（♡）: favorites テーブルに target_type = 'user' で登録
- 各職人クリック → CLI-006 へ遷移

### REQ-JS-008: ユーザー詳細 / 職人詳細（CLI-006）— 発注者専用

- 職人のプロフィール詳細を表示する
- 表示項目:
  - アイコン、氏名、年齢
  - バッジ: 本人確認済み、CCUS登録済み
  - PR動画（登録済みの場合）
  - 基本情報: 性別、都道府県、会社名/屋号
  - 自己紹介
  - 能力: 職種 × 経験年数、スキル、資格
  - 対応可能エリア
  - 空き日程（available_schedules）
  - 発注者からの評価（user_reviews の集計）
- 導線:
  - 「メッセージを送る」→ CLI-013（メッセージ詳細）
  - 「スカウトを送る」→ CLI-015（スカウト送信）
  - 「評価を見る」→ CLI-028（発注者評価）
- お気に入りボタン（♡）

## 非機能要件

### セキュリティ

- Middleware: ログイン済みユーザーのみアクセス可能。CLI-005, CLI-006 は発注者のみ
- RLS: 案件は status = 'open' のもののみ SELECT 可能（一般ユーザー向け）
- 応募制限は RLS ではなく Server Action で制御（RLSでのJOIN制御が複雑なため）
- 全フォーム入力に Zod バリデーション（クライアント + サーバー）

### パフォーマンス

- 案件一覧は RSC（サーバーコンポーネント）でデータ取得
- 検索フィルターは searchParams で管理
- ページネーションは RSC + searchParams

## 画面遷移
