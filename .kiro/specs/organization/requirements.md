# 組織管理機能（organization）— 要件定義

## 概要

法人プラン向けの機能。スカウトメッセージテンプレート管理、発注者プロフィール管理、担当者管理を提供する。

## 対象画面

| 画面ID | 画面名 | 概要 |
|--------|--------|------|
| CLI-016 | スカウトメッセージテンプレート一覧 | テンプレートのリスト表示 |
| CLI-017 | スカウトメッセージテンプレート詳細 | テンプレートの内容表示 |
| CLI-018 | スカウトメッセージテンプレート編集 | 既存テンプレートの編集 |
| CLI-019 | スカウトメッセージテンプレート新規作成 | 新規テンプレートの作成 |
| CLI-020 | 発注者情報詳細 | 会社情報・メッセージ・評判の表示 |
| CLI-021 | 発注者情報編集 | 会社情報の編集フォーム |
| CLI-022 | 担当者一覧 | 担当者リスト |
| CLI-023 | 担当者詳細 | 担当者の情報表示 |
| CLI-024 | 担当者編集 | 担当者情報の編集 |
| CLI-025 | 担当者新規作成 | 新規担当者の作成 |

## 対象ロール

- 発注者（Client）: テンプレート管理 + 発注者情報管理。担当者管理は法人プランのみ
- 管理責任者（Owner）: 全操作可能
- 管理者（Admin）: 管理者の作成以外は全操作可能
- 担当者（Staff）: テンプレート閲覧・利用は可。テンプレートの作成・編集・削除は不可。担当者管理画面へのアクセスは不可。発注者情報の閲覧は可だが編集は不可

## 機能要件

### スカウトテンプレート（CLI-016〜019）

#### REQ-ORG-001: テンプレート一覧（CLI-016）

- 自分（または所属組織）のスカウトテンプレートを一覧表示する
- 表示項目: タイトル、本文プレビュー、作成日
- 「新規作成」ボタン → CLI-019 へ遷移（担当者には非表示）
- 各テンプレートクリック → CLI-017 へ遷移
- 操作権限:
  - 管理責任者/管理者: 閲覧・新規作成可
  - 担当者: 閲覧のみ（新規作成ボタン非表示）

#### REQ-ORG-002: テンプレート詳細（CLI-017）

- テンプレートの全内容を表示する
- 表示項目: タイトル、本文、メモ
- 「編集する」ボタン → CLI-018 へ遷移（担当者には非表示）
- 「削除する」ボタン → 確認ポップアップ → 削除（担当者には非表示）
- 操作権限:
  - 管理責任者/管理者: 閲覧・編集・削除可
  - 担当者: 閲覧のみ（編集・削除ボタン非表示）

#### REQ-ORG-003: テンプレート編集（CLI-018）

- 既存テンプレートの内容を編集する
- 編集可能フィールド: タイトル、本文、メモ
- 保存成功時: CLI-017 へ遷移
- アクセス権限: 管理責任者/管理者のみ（担当者は Middleware でブロック）

#### REQ-ORG-004: テンプレート新規作成（CLI-019）

- 新規テンプレートを作成する
- 入力フィールド:
  - タイトル（必須）
  - 本文（必須）
  - メモ（任意）
- scout_templates テーブルにデータを保存
- 法人プランの場合: organization_id を自動設定（組織内で共有）
- 保存成功時: CLI-017 へ遷移
- アクセス権限: 管理責任者/管理者のみ（担当者は Middleware でブロック）

### 発注者プロフィール（CLI-020〜021）

#### REQ-ORG-005: 発注者情報詳細（CLI-020）

- 発注者（自社）のプロフィール情報を表示する
- 表示項目:
  - プロフィール画像、発注者表示名（会社名/氏名）
  - 募集職種、募集エリア、従業員規模
  - 求める働き方
  - 発注者メッセージ（紹介文）
  - 掲載中の案件数
  - 受注者からの評価サマリー（client_reviews の集計）
- 「編集する」ボタン → CLI-021 へ遷移（担当者には非表示）
- 「担当者一覧」導線 → CLI-022 へ遷移（法人プランかつ管理責任者/管理者のみ表示）
- 操作権限:
  - 管理責任者/管理者: 閲覧・編集可。担当者一覧への導線あり
  - 担当者: 閲覧のみ（編集ボタン・担当者一覧導線は非表示）

#### REQ-ORG-006: 発注者情報編集（CLI-021）

- アクセス権限: 管理責任者/管理者のみ（担当者は Middleware でブロック）
- 発注者プロフィールを編集する
- 編集可能フィールド:
  - プロフィール画像（JPEG/PNG、最大5MB）
  - 発注者表示名（必須）
  - 募集職種（複数選択、OptionSets）
  - 募集エリア（OptionSets）
  - 従業員規模（数値）
  - 求める働き方（OptionSets）
  - 発注者メッセージ（任意）
- client_profiles テーブルを更新
- 保存成功時: CLI-020 へ遷移

### 担当者管理（CLI-022〜025）— 法人プランのみ

#### REQ-ORG-007: 担当者一覧（CLI-022）

- 組織内の担当者一覧を表示する
- 表示項目: 氏名、メールアドレス、権限タグ（管理責任者/管理者/担当者）、代理アカウントバッジ
- ページネーション: 20件ずつ
- 「新規作成」ボタン → CLI-025 へ遷移
- 各担当者クリック → CLI-023 へ遷移
- 表示権限:
  - 管理責任者: 全担当者を表示
  - 管理者: 全担当者を表示
  - 担当者: アクセス不可（Middleware でブロック）

#### REQ-ORG-008: 担当者詳細（CLI-023）

- 担当者の情報を表示する
- 表示項目: 氏名、メールアドレス、権限、代理アカウントフラグ
- 「編集する」ボタン → CLI-024 へ遷移
- 「削除する」ボタン → 確認ポップアップ → 削除
  - 管理責任者は削除不可（内部管理者のみ変更可能）
  - 削除時の関連データ処理:
    - **organization_members**: 物理削除（組織のメンバー枠を空けるため）
    - **public.users**: そのまま残す（deleted_at を設定してソフトデリート）
    - **messages**: sender_id が削除対象の担当者になっている過去のメッセージはそのまま残す（過去のやり取りの記録として保持。送信者名は退会済みユーザー表示で対応）
    - **scout_templates**: owner_id が削除対象の担当者になっているテンプレートは、owner_id を組織の管理責任者（owner）に移譲する（テンプレートは会社の資産として残す）
  - 削除確認ポップアップに「この担当者が作成したテンプレートは管理責任者に引き継がれます」と表示
- 編集/削除権限:
  - 管理責任者: 管理者・担当者を編集/削除可能
  - 管理者: 担当者のみ編集/削除可能

#### REQ-ORG-009: 担当者編集（CLI-024）

- 担当者の情報を編集する
- 編集可能フィールド:
  - 管理責任者の場合: 名前・メールアドレスのみ編集可（権限変更不可）
  - 管理者/担当者の場合: 名前、メールアドレス、権限（管理者 or 担当者）
  - 代理アカウントチェックボックス（法人プランのみ、1法人1つ）
- 更新対象テーブル:
  - 名前・メールアドレスの変更 → users テーブルを更新（organization_members には名前・メールのカラムがないため）
  - 権限（org_role）の変更 → organization_members テーブルを更新
  - 代理アカウントフラグの変更 → organization_members テーブルを更新
- 保存成功時: CLI-023 へ遷移

#### REQ-ORG-010: 担当者新規作成（CLI-025）

- 新規担当者を作成する
- 入力フィールド:
  - 名前（必須）
  - メールアドレス（必須）
  - 権限（必須、管理者 or 担当者。管理責任者は内部管理者のみ作成可能）
  - 代理アカウント（チェックボックス、法人プランのみ、1法人1つ）
- フロー:
  1. 入力 → 確認画面
  2. 確認OK → Server Action 内で以下を実行:
     a. メールアドレスの重複チェック（auth.users に同じメールが既に存在しないか事前確認）
     b. `supabase.auth.admin.createUser()` で auth.users にアカウント作成（service_role キーを使用）
     c. auth.users 作成時のトリガーにより public.users が自動生成される
     d. public.users.role を 'staff' に更新
     e. organization_members テーブルに追加
  3. ステップ 2d〜2e はトランザクション（一連の処理をまとめて、途中で失敗したら全部取り消す仕組み）で実行する
  4. トランザクション失敗時のクリーンアップ: ステップ 2b が成功済みの場合は `supabase.auth.admin.deleteUser()` で auth.users からも削除する（幽霊アカウント防止）
  5. 会員登録メールを送信（パスワード設定用リンク）— メール送信失敗時は本体処理をロールバックしない（リトライ可能な設計とする）
- エラー時のユーザー表示:
  - メールアドレス重複: 「このメールアドレスは既に登録されています」
  - 人数制限超過: 「担当者の上限に達しています。プランのアップグレードをご検討ください」
  - その他のエラー: 「担当者の作成に失敗しました。時間をおいて再度お試しください」
- 人数制限:
  - 法人プラン（¥48,000）: 最大10人
  - 法人プラン高サポート（¥148,000）: 最大30人
  - 制限超過時: エラーメッセージ表示
- 代理アカウント制約: 1法人につき1つのみ

## 非機能要件

### セキュリティ

- Middleware: 発注者（課金済み）のみアクセス可能。担当者管理は法人プランのみ
- RLS:
  - テンプレート: 自分 or 所属組織のテンプレートのみ
  - 発注者情報: 自分の client_profiles のみ編集可能
  - 担当者: 所属組織のメンバーのみ閲覧/操作可能
- 権限階層の検証: Server Action で org_role を確認し、権限外の操作を拒否

## 画面遷移

```
CON-001 → CLI-016（テンプレ一覧）→ CLI-017（詳細）→ CLI-018（編集）
                                  → CLI-019（新規作成）

                                   → CLI-022（担当者一覧）→ CLI-023（担当者詳細）→ CLI-024（編集）
                                                         → CLI-025（新規作成）
```
